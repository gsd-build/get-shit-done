# Codebase Concerns

**Analysis Date:** 2025-07-15

## Tech Debt

**Massive install.js monolith (2,376 lines):**
- Issue: `bin/install.js` is a single 2,376-line file that handles installer logic for 4 different runtimes (Claude, OpenCode, Gemini, Codex), including format conversion, frontmatter parsing, uninstall, interactive prompts, and config manipulation.
- Files: `bin/install.js`
- Impact: Difficult to test, modify, or extend. Adding a 5th runtime would make it worse. Contains its own frontmatter parser separate from `get-shit-done/bin/lib/frontmatter.cjs`.
- Fix approach: Extract runtime-specific converters into separate modules under `bin/lib/`. Extract the duplicate frontmatter parser (`extractFrontmatterAndBody`, `extractFrontmatterField`) into shared code or reuse `get-shit-done/bin/lib/frontmatter.cjs`. Create a strategy pattern for runtime installation.

**Duplicate frontmatter parsing implementations:**
- Issue: `bin/install.js` has its own `extractFrontmatterAndBody()` and `extractFrontmatterField()` functions that are separate from the canonical `get-shit-done/bin/lib/frontmatter.cjs` module's `extractFrontmatter()`. They parse YAML differently and could diverge.
- Files: `bin/install.js` (lines 417–438), `get-shit-done/bin/lib/frontmatter.cjs` (lines 11–84)
- Impact: Bug fixes to frontmatter parsing must be applied in two places. The install.js version is simpler (single-field extraction) while the lib version handles full nested YAML.
- Fix approach: Refactor `bin/install.js` to import from `get-shit-done/bin/lib/frontmatter.cjs` or create a shared lightweight parser both can use.

**.github/ directory duplication:**
- Issue: The `.github/agents/` and `.github/get-shit-done/` directories contain copies of `agents/` and `get-shit-done/` that are installed artifacts (generated by the installer). However, they are gitignored, and all files diverge from source due to path rewrites. This is expected behavior for the GitHub Copilot runtime target, but the files exist in the repo working directory.
- Files: `.github/agents/gsd-*.agent.md` (11 files), `.github/get-shit-done/` (full tree mirror)
- Impact: Every agent and workflow file differs between source and `.github/` versions (path rewrites, tool name mappings, feature differences like `analysis_paralysis_guard`). Developers may accidentally edit the wrong copy. The `.gitignore` correctly excludes these installed copies.
- Fix approach: This is working as designed — the installer generates these. No code fix needed, but developers should be aware that `.github/` files are generated artifacts, not source.

**Command/workflow naming inconsistency:**
- Issue: The `commands/gsd/` directory (slash commands surfaced to users) and the `get-shit-done/workflows/` directory (internal workflow implementations) have overlapping but inconsistent naming. Several commands have different names than their workflow counterparts.
- Files: `commands/gsd/debug.md` → `get-shit-done/workflows/diagnose-issues.md`, `commands/gsd/resume-work.md` → `get-shit-done/workflows/resume-project.md`, `commands/gsd/reapply-patches.md` (no workflow equivalent), `commands/gsd/join-discord.md` (no workflow equivalent)
- Impact: Confusion when navigating the codebase. Workflows like `discovery-phase.md`, `execute-plan.md`, `transition.md`, `verify-phase.md` have no corresponding command entries.
- Fix approach: Audit and align naming between commands and workflows. Commands that lack workflow implementations may be simple redirects; workflows without commands may be internal-only (invoked by other workflows, not users).

**Backup file committed to repo:**
- Issue: `commands/gsd/new-project.md.bak` is a leftover backup file tracked by git.
- Files: `commands/gsd/new-project.md.bak`
- Impact: Minor clutter, but signals sloppy cleanup. Gets shipped to npm.
- Fix approach: Delete the file and add `*.bak` to `.gitignore`.

## Known Bugs

**`isGitIgnored` strips characters instead of quoting:**
- Symptoms: The `isGitIgnored()` function uses `targetPath.replace(/[^a-zA-Z0-9._\-/]/g, '')` to sanitize paths, which silently strips characters like spaces from paths rather than properly quoting them. A path like `.planning stuff/` would become `.planningstuff/` and fail silently.
- Files: `get-shit-done/bin/lib/core.cjs` (line 127)
- Trigger: Pass a path with spaces or special characters to `isGitIgnored()`.
- Workaround: In practice, `.planning` never has spaces, so this rarely triggers.

**`cmdCommit` reports "nothing_to_commit" for actual errors:**
- Symptoms: When `git commit` fails for reasons other than "nothing to commit" (e.g., hooks rejecting, GPG signing failure), the error is still reported as `reason: 'nothing_to_commit'` because both code paths use the same error object.
- Files: `get-shit-done/bin/lib/commands.cjs` (lines 246–255)
- Trigger: Have a pre-commit hook that rejects the commit.
- Workaround: None — the error is silently misclassified.

**`gsd-check-update.js` hardcodes `~/.claude` paths:**
- Symptoms: The update check hook hardcodes `~/.claude/cache/` and `~/.claude/get-shit-done/VERSION` paths, which are only correct for Claude Code. If installed for OpenCode or Gemini, the hook checks the wrong version file.
- Files: `hooks/gsd-check-update.js` (lines 12–17)
- Trigger: Install GSD globally for OpenCode or Gemini and observe stale update cache.
- Workaround: The installer templating system replaces `~/.claude/` paths during installation, so the runtime hook should have correct paths. The source file is the problem only if run directly.

## Security Considerations

**Shell command injection surface in `execGit`:**
- Risk: `execGit()` constructs shell commands via string concatenation (`'git ' + escaped.join(' ')`). While arguments are escaped with single-quote wrapping, the function uses `execSync()` with a shell, not `execFileSync()`. Edge cases in the quoting logic could allow injection.
- Files: `get-shit-done/bin/lib/core.cjs` (lines 137–156)
- Current mitigation: Arguments are single-quoted with internal quotes escaped. Safe chars bypass quoting. The function is only called with internally-generated arguments, never with raw user input from stdin.
- Recommendations: Migrate to `execFileSync('git', args)` which bypasses the shell entirely and eliminates the injection surface. This is a low-risk but clean improvement.

**`isGitIgnored` uses string-concatenated shell command:**
- Risk: Same pattern as `execGit` — constructs command via string concatenation rather than array-based execution. The sanitization regex strips rather than rejects bad characters.
- Files: `get-shit-done/bin/lib/core.cjs` (line 127)
- Current mitigation: Only called with hardcoded path strings (`.planning`), not user input.
- Recommendations: Refactor to use `execFileSync('git', ['check-ignore', '-q', '--', targetPath])`.

**`init.cjs` uses unescaped `find` shell command:**
- Risk: `cmdInitNewProject()` calls `execSync('find . -maxdepth 3 ...')` with shell expansion. While the command is hardcoded (no user input), the shell invocation is unnecessary and platform-dependent (fails on Windows).
- Files: `get-shit-done/bin/lib/init.cjs` (line 174)
- Current mitigation: Wrapped in try/catch with empty handler; failure is silent.
- Recommendations: Replace with `fs.readdirSync()` recursive walk — already used elsewhere in the codebase.

## Performance Bottlenecks

**Synchronous file I/O throughout entire CLI:**
- Problem: Every file operation uses `fs.readFileSync`, `fs.writeFileSync`, `fs.readdirSync`, etc. The CLI performs many file reads per command invocation (config, state, roadmap, phase directories, plans, summaries).
- Files: All files under `get-shit-done/bin/lib/*.cjs`
- Cause: The CLI was designed for simplicity with synchronous I/O. Each `gsd-tools.cjs` invocation is a short-lived process, so async I/O would add complexity without meaningful user-facing benefit.
- Improvement path: Low priority. The CLI exits quickly (~50-300ms per invocation). Only worth addressing if commands grow to scan many hundreds of phase directories.

**`cmdHistoryDigest` scans all archived milestones:**
- Problem: Reads every SUMMARY.md across all phases and all archived milestones. For projects with many milestones (>5) and many phases, this becomes a full disk scan.
- Files: `get-shit-done/bin/lib/commands.cjs` (lines 99–197)
- Cause: No caching — rebuilds digest from scratch on every invocation.
- Improvement path: Cache digest to `.planning/cache/history-digest.json` and invalidate when phase directories change (compare mtimes).

**`cmdRoadmapAnalyze` reads phase directories twice:**
- Problem: For each phase found in ROADMAP.md, the function reads the corresponding phase directory to count plans and summaries. This duplicates work already done by `searchPhaseInDir` in `core.cjs`.
- Files: `get-shit-done/bin/lib/roadmap.cjs` (lines 93–218)
- Cause: The function was written independently of the existing phase-scanning infrastructure.
- Improvement path: Refactor to use `findPhaseInternal()` which already collects plan/summary counts.

## Fragile Areas

**Custom YAML frontmatter parser:**
- Files: `get-shit-done/bin/lib/frontmatter.cjs` (lines 11–84)
- Why fragile: The `extractFrontmatter()` function is a hand-rolled YAML parser using a stack-based approach. It handles basic key-value pairs, arrays, and nested objects but has known limitations: no support for multi-line strings, block scalars (`|`, `>`), complex YAML types, or comments. The indent-based stack tracking is sensitive to mixed tabs/spaces.
- Safe modification: Add tests for any new YAML pattern before modifying. The test suite in `tests/frontmatter.test.cjs` covers current patterns well.
- Test coverage: 353 lines of tests — reasonably covered for current use cases.

**STATE.md regex-based field extraction:**
- Files: `get-shit-done/bin/lib/state.cjs` (lines 153–167, 407–506, 515–621)
- Why fragile: Three separate functions (`stateExtractField`, `cmdStateSnapshot`, `buildStateFrontmatter`) all independently parse STATE.md using regex patterns like `/\*\*FieldName:\*\*\s*(.+)/i`. If the STATE.md template format changes (e.g., heading levels, bold syntax), all three break independently. The `writeStateMd` function syncs a frontmatter cache on every write, creating a dual-representation of state.
- Safe modification: Always update all three extraction points together. Run `tests/state.test.cjs` after any change. The `writeStateMd` wrapper ensures frontmatter stays in sync with body.
- Test coverage: 1,254 lines of tests — well covered.

**Roadmap markdown parsing:**
- Files: `get-shit-done/bin/lib/roadmap.cjs`, `get-shit-done/bin/lib/core.cjs` (lines 320–352)
- Why fragile: ROADMAP.md is parsed via regex to extract phase headings, goals, success criteria, dependency info, and checkbox status. The regex patterns assume specific heading levels (`#{2,4}`), specific bold-field patterns (`**Goal:**`), and specific checkbox syntax (`- [x]`). Any deviation in ROADMAP.md formatting causes silent parse failures.
- Safe modification: Always test against sample ROADMAP.md files with varying formats. The `tests/roadmap.test.cjs` (670 lines) provides good coverage.
- Test coverage: Good, but edge cases around malformed roadmaps may exist.

**Phase directory naming conventions:**
- Files: `get-shit-done/bin/lib/core.cjs` (lines 164–199, 201–244)
- Why fragile: Phase directories must follow the pattern `NN-name` or `NNA-name` or `NN.N-name`. The `normalizePhaseName`, `comparePhaseNum`, and `searchPhaseInDir` functions implement complex parsing for integer phases, letter-suffixed phases (12A), and decimal phases (12.1). Adding a new numbering scheme requires updating all three functions plus their callers.
- Safe modification: The test suite in `tests/phase.test.cjs` (1,463 lines) is comprehensive. Run full suite after any change to phase naming.
- Test coverage: Excellent — highest test line count in the project.

## Scaling Limits

**Markdown-as-database pattern:**
- Current capacity: Works well for projects with 1-3 milestones, ~20 phases, ~50 plans.
- Limit: STATE.md, ROADMAP.md, and REQUIREMENTS.md are parsed via regex on every CLI invocation. With 100+ phases or very large roadmaps (>5,000 lines), regex parsing becomes slower and more error-prone.
- Scaling path: The `state json` command and `writeStateMd` frontmatter sync are a step toward structured data. A future migration could use `.planning/state.json` as the authoritative source with markdown rendering for human readability.

**No file locking for concurrent writes:**
- Current capacity: Works when a single AI agent session operates at a time.
- Limit: If two parallel agents attempt to update STATE.md simultaneously (possible with the `parallelization: true` config), one write may overwrite the other. The `writeStateMd` function reads, transforms, and writes without any locking mechanism.
- Scaling path: Implement advisory file locking (e.g., `fs-ext` flock or `.lock` sentinel files) around STATE.md writes, or move to an append-only log for state changes.

## Dependencies at Risk

**No runtime dependencies — CLI is zero-dependency:**
- The `get-shit-done/bin/` code uses only Node.js built-ins (`fs`, `path`, `child_process`, `os`). This is a significant strength — no supply chain risk for runtime.
- `bin/install.js` similarly uses only built-ins.
- Dev dependencies (`c8`, `esbuild`) are build/test only and not shipped.
- Risk level: Very low.

**Node.js version floor (>=16.7.0):**
- Risk: The codebase uses `node:test` for testing (available since Node 18) and modern patterns like `catch {}` (parameterless catch, Node 16+). The `engines` field specifies `>=16.7.0` but test infrastructure requires Node 18+.
- Files: `package.json` (engines field), `tests/*.test.cjs` (all use `require('node:test')`)
- Impact: Tests won't run on Node 16 despite the engine claim. The runtime code itself should work on Node 16.7+.
- Migration plan: Bump `engines` to `>=18.0.0` to match actual test requirements, or document that Node 16 is runtime-only compatible.

## Missing Critical Features

**No input validation on CLI arguments:**
- Problem: `gsd-tools.cjs` routes commands via a simple `switch` statement with positional argument access (`args[1]`, `args[2]`). There is no schema validation, type checking, or helpful error messages for malformed arguments. Passing the wrong number of args often causes cryptic `TypeError` or `undefined` errors deep in command handlers.
- Files: `get-shit-done/bin/gsd-tools.cjs` (lines 178–585)
- Blocks: Better error messages, argument completion, self-documenting CLI.

**No integration tests for the installer:**
- Problem: `bin/install.js` (2,376 lines) has no automated tests. The `tests/codex-config.test.cjs` tests only the Codex config TOML generation functions exported from install.js. The installer's interactive flow, multi-runtime conversion, uninstall, and settings.json manipulation are untested.
- Files: `bin/install.js`, `tests/codex-config.test.cjs` (412 lines — Codex only)
- Blocks: Confident refactoring of the installer. Regressions in Gemini/OpenCode/Claude installation are only caught by manual testing.

## Test Coverage Gaps

**Installer (`bin/install.js`) almost entirely untested:**
- What's not tested: Interactive prompts, file copy/conversion for Claude/OpenCode/Gemini, uninstall flow, settings.json manipulation, hook registration, orphaned file cleanup, `processAttribution()`, `convertClaudeToOpencodeFrontmatter()`, `convertClaudeToGeminiAgent()`.
- Files: `bin/install.js` (2,376 lines), `tests/codex-config.test.cjs` (covers ~100 lines of Codex-specific code)
- Risk: A regression in the installer could break the primary user experience (install flow). The 4-runtime conversion matrix is complex and has many edge cases.
- Priority: High — this is the user's first interaction with the tool.

**Hooks not tested:**
- What's not tested: `hooks/gsd-context-monitor.js`, `hooks/gsd-statusline.js`, `hooks/gsd-check-update.js` — all three hooks have zero test coverage.
- Files: `hooks/gsd-context-monitor.js` (122 lines), `hooks/gsd-statusline.js` (108 lines), `hooks/gsd-check-update.js` (63 lines)
- Risk: Hooks are invoked by Claude Code on every tool use (context monitor) or session start (update check). A crash in a hook blocks or degrades the AI session. All three have silent `catch` blocks that mask bugs.
- Priority: Medium — hooks are simple but critical for user experience.

**`init.cjs` compound commands have limited testing:**
- What's not tested: `tests/init.test.cjs` (849 lines) covers the basic init commands, but the compound `cmdInitExecutePhase`, `cmdInitPlanPhase`, etc. aggregate many data sources and the interaction between them (config + state + roadmap + phase dir) has combinatorial edge cases.
- Files: `get-shit-done/bin/lib/init.cjs` (710 lines)
- Risk: The init commands produce JSON payloads consumed by workflow markdown files. A wrong field name or null value causes workflow failures that are hard to debug.
- Priority: Medium — these are upstream of all workflows.

**57 empty catch blocks throughout the codebase:**
- What's not tested: Error recovery paths. Every `catch {}` block represents a code path where errors are silently swallowed, making it impossible to know if the "no data" response is correct or a masked bug.
- Files: Distributed across all `get-shit-done/bin/lib/*.cjs` files — 39 in lib code, additional in `bin/install.js` and `hooks/`
- Risk: Silent failures that produce confusing downstream behavior. For example, if `fs.readdirSync` throws because of a permissions issue, the command returns empty arrays as if no phases exist.
- Priority: Medium — add at minimum `console.error` or structured error tracking to distinguish "no data" from "access error."
