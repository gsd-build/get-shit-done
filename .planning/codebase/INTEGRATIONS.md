# External Integrations

**Analysis Date:** 2026-02-05

## APIs & External Services

**AI IDE Runtimes:** (Primary integration targets)
- Claude Code - Anthropic's native IDE with custom agent support
  - Integration method: Command definitions in `.claude/commands/gsd/`, agent specs in `.claude/agents/`
  - Authentication: Uses user's existing Claude Code authentication

- OpenCode - Open source AI IDE alternative with experimental agent support
  - Integration method: Command definitions in `.config/opencode/commands/gsd/`, requires `experimental_agents` enabled
  - Authentication: Uses user's OpenCode authentication

- Gemini CLI - Google's command-line AI interface with experimental agent support
  - Integration method: Command definitions in `.gemini/commands/gsd/`, YAML-formatted tools array
  - Authentication: Uses user's Gemini CLI authentication

**No External Service APIs:** The system does not directly integrate with external APIs (no Stripe, Slack, GitHub API, database services, etc.). All external service communication happens through the AI IDE runtimes and their provided APIs.

## Data Storage

**Databases:**
- Not applicable. GSD does not manage databases. Users' target projects may use databases, but GSD itself is stateless.

**File Storage:**
- Local filesystem only
- Configuration stored in user's IDE config directories:
  - `~/.claude/get-shit-done/` (Claude Code global)
  - `./.claude/get-shit-done/` (Claude Code local project)
  - `~/.config/opencode/get-shit-done/` (OpenCode global, XDG compliant)
  - `~/.gemini/get-shit-done/` (Gemini CLI global)
- Project planning state stored in `.planning/` directory within user's project

**Caching:**
- Update check cache: `~/.claude/cache/gsd-update-check.json` (generated by `hooks/gsd-check-update.js`)
- Session cache: Stored in user's IDE config cache directory
- No external caching service

## Authentication & Identity

**Auth Provider:**
- Custom (federated through IDE runtimes)
- Implementation approach:
  - GSD does not handle authentication directly
  - Authentication delegated to AI IDE (Claude Code, OpenCode, or Gemini CLI)
  - Each IDE handles user identity, API keys, and session management
  - GSD reads IDE configuration paths via environment variables or XDG Base Directory standard

**User Configuration:**
- Installation creates IDE-specific command and agent files in user's local config
- User selects which IDE(s) to install to during `npx get-shit-done-cc` setup
- Separate installations for each IDE (Claude, OpenCode, Gemini) can coexist

## Monitoring & Observability

**Error Tracking:**
- Not integrated. Errors handled at CLI level in `bin/install.js` and hooks with console output
- No remote error reporting

**Logs:**
- Console output only (stdout/stderr)
- Verbose logging available through terminal output when running commands
- Git hook logging: Status updates printed to terminal via `hooks/gsd-statusline.js`
- Update check results cached locally in `~/.claude/cache/gsd-update-check.json`

## CI/CD & Deployment

**CI Pipeline:**

- Actual CI config not in scope of this codebase analysis

**Deployment Process:**

- Installation: `npm install -g get-shit-done-cc` (global) or `npm install` (local)

## Version Management

**Version Tracking:**
- Package version: `package.json` contains `"version": "1.11.1"`
- VERSION file: Created in installation directory at `~/.claude/get-shit-done/VERSION` (or equivalent for OpenCode/Gemini)
- Update checking: `hooks/gsd-check-update.js` spawns background process to check npm for newer versions
- Update notification: Shown in IDE status line if newer version available

**Version Check Implementation:**
- File: `hooks/gsd-check-update.js` - Spawned as background process
- Checks: Compares local VERSION file against npm latest tag
- Cache: Results cached in `~/.claude/cache/gsd-update-check.json` to avoid repeated checks within session
- User notification: Displayed in Claude Code statusline when update available

## Environment Configuration

**Required env vars:**
- None required for basic operation
- Optional environment variables (for advanced configuration):
  - `CLAUDE_CONFIG_DIR` - Custom location for Claude Code config (default: `~/.claude/`)
  - `OPENCODE_CONFIG_DIR` or `OPENCODE_CONFIG` - Custom location for OpenCode config
  - `XDG_CONFIG_HOME` - For XDG Base Directory standard (OpenCode default: `$XDG_CONFIG_HOME/opencode`)
  - `GEMINI_CONFIG_DIR` - Custom location for Gemini CLI config

**Secrets location:**
- API keys and authentication tokens belong to AI IDE runtimes (Claude Code, OpenCode, Gemini)
- GSD does not store or manage secrets
- User credentials for AI services stored in respective IDE configuration (outside GSD's scope)

## Git Integration

**Git Hooks:**
- Installation location: `.git/hooks/` in user's project
- Hook files:
  - `pre-commit`: Runs `hooks/gsd-check-update.js` in background
  - `post-commit`: Runs `hooks/gsd-precompact.js` if git-slim available
  - These hooks copied during IDE setup to monitor and optimize GSD usage

**Git Operations:**
- GSD orchestrates git commits through executor agents
- Agents use `git add`, `git commit`, `git status`, `git log`, `git diff` commands
- GSD respects `.gitignore` - `.planning/` directory can be gitignored
- Version control hooks set up to track GSD update availability

## Webhooks & Callbacks

**Incoming:**
- Not applicable. GSD is a local CLI tool, does not expose webhooks or HTTP endpoints.

**Outgoing:**
- Not applicable. GSD does not make outbound HTTP requests to external services.
- Update checks: Local file system only (no phone-home to check versions beyond npm registry via standard npm commands)

## IDE Integration Points

**Command System:**
- Commands are registered as markdown files in `commands/gsd/{command-name}.md`
- Each command file contains YAML frontmatter with:
  - `name`: Command identifier (e.g., `gsd:map-codebase`)
  - `description`: Help text
  - `argument-hint`: Usage documentation
  - `allowed-tools`: Which tools agent can use

**Agent System:**
- Agents are specification documents in `agents/gsd-*.md`
- Agents follow standardized YAML structure with:
  - `name`: Agent identifier
  - `description`: Purpose
  - `tools`: Allowed tools (Read, Write, Bash, Grep, Glob, etc.)
  - `color`: Terminal color for output
  - YAML-embedded role descriptions and process flows

**Custom Tools:**
- GSD does not define custom tools
- Uses standard IDE tools available to all agents: Read, Write, Edit, Bash, Grep, Glob, Task, AskUserQuestion
