# Phase 18.2: Resolve add-new-feature end-to-end workflow blockers from labs QA - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix end-to-end workflow blockers preventing "add new feature" from completing in the UI for GSD-initialized projects. Scope includes: add-plan creation entry path, discuss progression with default acceptance, plan/verify state binding, verify approval gating, and `todo` vs `todo-app` parity issues that currently block flow completion.

This phase clarifies implementation of existing workflow behavior. It does not expand product scope beyond resolving the reported blockers.

</domain>

<decisions>
## Implementation Decisions

### Add Plan Entry and Category UX
- `New Plan` entry point lives in the global project header across all workflow tabs.
- Plan creation uses a modal with prefilled quick defaults:
  - title
  - category
  - intent
- Category source is hybrid:
  - fixed base category set
  - project-custom categories
- Category behavior:
  - auto-detect from user intent
  - show detected category in modal before submit
  - allow manual override via dropdown
  - block submit until category is explicitly confirmed
  - if detection confidence is low, preselect `general` and require user choice/confirmation

### Project/API Parity and Reliability Contract (MVP)
- Only GSD-initialized projects are considered supported in the dashboard/workflow surface.
- Dashboard/search should include direct-linkable initialized projects (e.g., `todo`) even when discovery metadata is incomplete.
- If an expected workflow endpoint is unavailable in MVP, UI should show explicit error (no silent fallback).
- Endpoint unavailability should be treated as workflow precondition/state issue for initialized projects (for example, phase cannot execute yet because upstream state like plan generation is missing), not as capability negotiation.
- Add TODO for post-MVP resilience improvements around endpoint handling.

### Discuss Progression with "Accept All Defaults"
- Discuss UI presents options in ranked order with recommended-first labeling:
  - `Option 1 ... (Recommended)`, `Option 2`, ..., `Option N`, `Accept all defaults`
- "Accept all defaults" is an explicit button in the options panel (not free-text intent detection).
- Clicking "Accept all defaults" applies the top recommended option for each unresolved question in the current phase.
- Audit trail requirements:
  - summary event (`defaults accepted`)
  - per-decision records for each defaulted choice

### Plan/Verify Binding and Action Gating
- Plan rendering rule: authoritative server payload overrides stale local empty-state.
  - If backend returns generated tasks, UI must render tasks even if local store indicates none.
- Reject-with-gaps flow:
  - redirected Plan page auto-focuses a "Fix Plans from selected gaps" section
  - selected gaps are preselected and visible as chips/tokens
- Approval gating:
  - Approve action is disabled until verify run reaches terminal status (`completed` or `failed`) and results count is greater than zero.
- Verify re-run behavior:
  - keep prior results visible but mark as stale while new run is in progress
  - replace with new run results on completion

### Claude's Discretion
- Exact visual styling for modal, option list, stale banners, and disabled states.
- Exact confidence threshold implementation details for category auto-detection.
- Exact wording of error messages and audit event copy, provided behavior contracts above are preserved.

</decisions>

<specifics>
## Specific Ideas

- Option ordering in Discuss must visibly prioritize recommended choices and provide a final "Accept all defaults" action.
- "Accept all defaults" should be deterministic: choose recommended option for each unresolved prompt in that phase.
- Plan page should not show "No plan tasks" when backend payload contains tasks.
- Verify should preserve previous results with stale labeling during a re-run rather than clearing to blank.

</specifics>

<deferred>
## Deferred Ideas

- Post-MVP endpoint resilience model (for example, richer capability signaling/fallback behavior beyond explicit error display).
- Broader workflow support beyond GSD-initialized projects.

</deferred>
