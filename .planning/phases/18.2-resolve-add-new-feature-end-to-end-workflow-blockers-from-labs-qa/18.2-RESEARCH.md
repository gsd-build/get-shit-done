**Date:** 2026-03-14  
**Status:** Complete

## Objective

Define an implementation approach to restore a complete add-new-feature flow for GSD-initialized projects across Discuss -> Plan -> Verify, while honoring locked context decisions for category confirmation, explicit error handling, and action gating.

## Current-State Findings

1. Project-level UI has no shared workflow action header and no Add/New Plan affordance.
2. Discuss page is free-text chat only; no ranked option panel or explicit "Accept all defaults" control.
3. Plan page relies on local `planData` and can render empty-state despite valid backend payload timing/state races.
4. Verify page approval controls are not explicitly tied to run terminal status + result count; stale-result handling for reruns is not modeled.
5. Project/API reliability mismatches can surface as raw 404 behavior with limited user-level workflow guidance.

## Implementation Strategy

### 1) Establish Project/API Contract for Initialized Projects

- Enforce "GSD-initialized only" support in discovery/surface logic.
- Include direct-linkable initialized projects in dashboard/search even when metadata is incomplete.
- Normalize workflow endpoint errors into explicit UI-facing failure states for MVP (no silent fallback).
- Keep a post-MVP TODO path for richer resilience/capability signaling.

### 2) Add Shared New Plan Entry + Category Confirmation Flow

- Introduce a shared workflow header across project tabs with `New Plan`.
- Implement modal-based creation flow with prefilled defaults: title, category, intent.
- Add category detection utility (intent -> suggested category) with confidence output.
- Require explicit category confirmation before submit:
  - high confidence: preselect suggestion, still require explicit confirmation
  - low confidence: preselect `general`, require user choice/confirmation
- Source categories from hybrid model:
  - base system categories
  - project-scoped custom categories

### 3) Discuss Defaults UX with Deterministic Behavior

- Render ranked options for each discuss decision, with recommended option first.
- Add explicit action row ending with `Accept all defaults`.
- Accept-all applies recommended option for each unresolved decision in current phase.
- Persist audit trail at two levels:
  - summary event (`defaults accepted`)
  - per-decision applied option records

### 4) Plan Binding + Gap-Fix Visibility

- Treat server payload as authoritative when present.
- Replace stale local empty-state if backend returns tasks.
- On reject-with-gaps redirect, auto-focus a fix-plan section with selected gap chips preloaded.

### 5) Verify Gating + Stale Results During Reruns

- Disable approval until:
  - run status is terminal (`completed` or `failed`)
  - result count > 0
- During rerun:
  - keep previous results visible
  - mark as stale with clear in-progress banner
  - atomically swap to new results on completion

## Risks and Mitigations

1. **Risk:** Client and server status semantics diverge.
   - **Mitigation:** centralize status predicates in shared UI helpers/store selectors.
2. **Risk:** Discuss accept-default path applies wrong choices.
   - **Mitigation:** deterministic mapping by decision ID + recommended rank index.
3. **Risk:** Category confidence introduces inconsistent UX.
   - **Mitigation:** fixed threshold constant, deterministic fallback to `general`.
4. **Risk:** Endpoint precondition errors become noisy.
   - **Mitigation:** standardized error copy and actionable next-step hints.

## Test Strategy

- Unit tests:
  - category detection confidence + fallback rules
  - verify approval gating predicate
  - stale-result marker transitions
- Component tests:
  - New Plan modal confirmation constraints
  - Discuss option panel + accept-all behavior
  - Plan fix-gap preselection and focus
- E2E tests:
  - add new plan from global header on multiple tabs
  - discuss accept defaults progression
  - plan page renders tasks when backend payload exists
  - verify approve disabled while running; enabled only at terminal+results

## Plan Shape Recommendation

- Total plans: 5
- Wave model:
  - Wave 1: API/project contract foundation + New Plan/category modal foundation
  - Wave 2: Discuss defaults UX + Plan state/fix-gap flow
  - Wave 3: Verify gating/stale rerun behavior + full E2E closure

## Out of Scope (Deferred)

- Capability negotiation endpoint and dynamic route probing (post-MVP).
- Support for non-GSD-initialized project types.
