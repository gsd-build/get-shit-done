  <files>get-shit-done/workflows/discuss-phase.md</files>
  <action>
Add new step `extract_from_docs` after `parse_docs_flag`. This step runs only when `has_docs=true`.

Step responsibilities:
1. For each document in valid_docs:
   - Read the document using Read tool
   - If large (>1000 lines), use focused search based on gray areas
   - Otherwise read entire document

2. Identify gray areas relevant to the phase (reuse analyze_phase logic)

3. For each gray area, search documents and classify findings:
   - **Explicit** — Direct statement with quote. Notation: `[from: filename.md, "Section Name"]`
   - **Inferred** — Reasonable conclusion with supporting evidence. Notation: `[inferred from: filename.md, "Section Name"]`. Only when evidence is strong.
   - **Ambiguous** — Conflicting information found. Notation: `[ambiguous: file1.md says X, file2.md says Y]`. Present both signals.
   - **Gap** — Nothing found. Notation: `[gap - needs user input]`

4. Compile extraction_results:
```markdown
### Extracted Decisions

**From prd.md:**
| Gray Area | Finding | Classification | Location |
|-----------|---------|----------------|----------|
| Layout | "Use card-based layout" | Explicit | UI Requirements |
| Loading | "smooth experience" | Inferred | Performance section |

**From spec.md:**
| Gray Area | Finding | Classification | Location |
|-----------|---------|----------------|----------|
| Loading | "Paginate 20 items" | Explicit | Data Loading |
```

5. Detect conflicts (same gray area, different answers from different docs or within same doc) - mark as Ambiguous

6. Store extraction_results for present_extractions step

Insert step XML after parse_docs_flag:
```xml
<step name="extract_from_docs">
**Requires:** has_docs=true, valid_docs list from parse_docs_flag

For each document in valid_docs:

1. Read the document:
   - If large (> 1000 lines), use focused search based on phase gray areas
   - Otherwise, read entire document

2. Identify phase gray areas (same analysis as analyze_phase step)

3. For each gray area, search for relevant content and classify:

**Classification tiers:**
- **Explicit** — Direct statement with quotable text
  - Notation: `[from: filename.md, "Section Name"]`
- **Inferred** — Reasonable conclusion from context
  - Only when evidence is strong
  - Notation: `[inferred from: filename.md, "Section Name"]`
- **Ambiguous** — Conflicting information found
  - Present both signals with sources
  - Notation: `[ambiguous: file1.md says X, file2.md says Y]`
- **Gap** — No coverage found for this area
  - Route to standard questioning
  - Notation: `[gap - needs user input]`

4. Compile findings into extraction_results structure

5. Detect conflicts (same area, different answers) - mark as Ambiguous

Store extraction_results for present_extractions step.
Proceed to present_extractions.
</step>
```

<task type="auto">
  <name>Task 2: Add present_extractions step</name>
  <files>get-shit-done/workflows/discuss-phase.md</files>
  <action>
Add new step `present_extractions` after `extract_from_docs`. This step runs only when `has_docs=true`.

Step responsibilities:
1. Group extraction_results by resolution status:
   - **Resolved:** Explicit + Inferred decisions
   - **Needs Resolution:** Ambiguous (conflicts)
   - **Gaps:** Areas with no coverage

2. Present grouped summary to user:
```markdown
## Decisions Extracted from Documents

### [Area 1] (from prd.md)
- Decision text [Explicit]
- Another decision [Inferred from visual design section]

### [Area 2] (from prd.md + spec.md)
- Decision [Explicit, prd.md]

---

### Conflicts Found

**[Area with conflict]:**
- prd.md: "infinite scrolling"
- spec.md: "paginate with 20 items"

Which approach should we use?

---

### Areas Not Covered by Documents

- Empty state design
- Error handling
```

3. Use AskUserQuestion to get user response:
   - header: "Review"
   - question: "Review the extracted decisions above. Correct anything, then we'll discuss the gaps."
   - options:
     - "Looks good, discuss gaps" - proceed to discuss_areas for gap areas only
     - "I want to change something" - allow inline overrides

4. If user wants changes:
   - For each override, note: "Doc said X, user chose Y [override]"
   - Update extraction_results with user_overrides

5. If all areas resolved (no gaps, no ambiguous):
   - question: "Documents resolved all gray areas. Create CONTEXT.md?"
   - options: "Yes, create it" / "I want to discuss something first"

6. Route unresolved areas (gaps + ambiguous after user input) to discuss_areas step
   - Set resolved_areas from extraction (skip in discuss_areas)
   - Set pending_areas = gaps + unresolved ambiguous

Insert step XML after extract_from_docs:
```xml
<step name="present_extractions">
**Requires:** extraction_results from extract_from_docs

Group by resolution status:

**Resolved (Explicit + Inferred):**
```
## Decisions Extracted from Documents

### [Area] (from source.md)
- Decision [Classification]
```

**Needs Resolution (Ambiguous):**
```
### Conflicts Found

**[Area]:**
- file1.md: "option A"
- file2.md: "option B"

Which approach?
```

**Gaps:**
```
### Areas Not Covered by Documents

- [gap area 1]
- [gap area 2]
```

Use AskUserQuestion:
- header: "Review"

  - "Looks good, discuss gaps" - route gaps to discuss_areas
  - "I want to change something" - allow inline overrides

**If user overrides:**
For each override: "Doc said X, user chose Y [override]"
Update extraction_results with user_overrides.

**If all areas resolved (no gaps, no ambiguous):**
- question: "Documents resolved all gray areas. Create CONTEXT.md?"
- options: "Yes, create it" / "I want to discuss something first"

**Routing:**
- resolved_areas = Explicit + Inferred + user-resolved Ambiguous (skip in discuss_areas)
- pending_areas = gaps + remaining ambiguous (route to discuss_areas)

If pending_areas empty and user confirms: proceed to write_context.
Otherwise: proceed to discuss_areas with pending_areas only.
</step>
```
