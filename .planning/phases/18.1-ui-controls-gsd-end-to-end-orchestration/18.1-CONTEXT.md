# Phase 18.1: UI controls GSD end-to-end orchestration - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Define and deliver the UI control model for end-to-end orchestration across GSD workflows. This phase covers where orchestration controls live, how run lifecycle actions behave, concurrency/parallelism governance, and safety/recovery UX. It does not add new workflow capabilities beyond control, visibility, and gating of existing orchestration flows.

</domain>

<decisions>
## Implementation Decisions

### Control surface placement
- Use a persistent project-level top control bar across `Discuss`, `Plan`, `Execute`, and `Verify`.
- Keep a compact per-page status strip that shows current run status and limited quick actions.
- Dashboard project cards should show orchestration status only; no destructive or advanced actions from cards.
- Click UI is primary. Command palette/shortcuts are secondary entry points.

### Run lifecycle controls
- v1 must include all run actions: `Start`, `Pause`, `Resume`, `Abort`, `Retry failed step`, `Re-run from checkpoint`.
- `Abort` must support both modes: `Stop now` and `Graceful stop at safe checkpoint`.
- Failure handling should support both paths: `Retry` and `Open fix plan`.
- Plan task editing is locked while a run is active. User must pause, edit, then resume; editing re-locks on resume.

### Scope of control and parallelism
- Default start behavior should execute all plans with wave semantics: parallel where eligible, sequential where dependencies require it.
- Multiple active runs are allowed only when a dependency/parallelism check confirms no conflicts.
- Cross-project concurrency is allowed.
- Starting additional runs in the same project requires a check-parallelism function; dependency-conflicting starts are blocked.
- UI must include a workflow diagram showing sequential edges, parallel task groups, active runs, and which phases/runs have been assessed for parallelism.

### Safety and recovery UX
- Confirmation is required for all major control actions: `Abort`, `Stop now`, `Cancel current and start new`, `Re-run from checkpoint`, and `Retry`.
- Dependency-check failures should show expandable details (not just a simple block banner).
- Pause duration is indefinite; paused jobs must be visible on the main dashboard.
- Resuming after plan edits requires full revalidation before continuation.

### Claude's Discretion
- Exact visual styling, iconography, and component composition of control bar/strips/diagram.
- Exact copy text for confirmations, warnings, and status badges.
- Presentation details for revalidation summaries and dependency conflict detail expansion.

</decisions>

<specifics>
## Specific Ideas

- User explicitly wants a visual workflow diagram for orchestration parallelism: show parallel tasks vs sequential phases and explicit “assessed for parallelism” status.
- Editing policy is strict: no live edits during active execution; pause-edit-resume only.

</specifics>
