# Phase 18.1: UI controls GSD end-to-end orchestration - Research

**Researched:** 2026-03-14
**Domain:** Cross-workflow orchestration controls, lifecycle gating, dependency-aware parallelism, and recovery UX
**Confidence:** HIGH

## Summary

Phase 18.1 should establish a single orchestration control model that spans Discuss, Plan, Execute, and Verify, while preserving dependency safety and enabling parallel eligible runs. The key implementation need is a project-level control plane that is visible on all workflow pages and coordinated by a shared store + server-side orchestration policy checks.

The existing codebase already has strong primitives to build on:
- Web has dedicated stores for plan/execution/verification and hooks for socket/control flows.
- Server orchestrator already tracks agent status and checkpoint state, including pause semantics.
- Project APIs already expose high-level status and can be extended for paused/parallelism details.

The phase should focus on four concrete capabilities from CONTEXT.md:
1. Shared control surface and compact page strips.
2. Full lifecycle actions with confirmation and edit-locking policy.
3. Dependency-aware parallelism gating with workflow diagram visibility.
4. Recovery and safety UX, including full revalidation on resume-after-edit and dashboard paused visibility.

## Codebase Findings

### Existing assets to reuse
- `apps/web/src/hooks/useAgentControl.ts` already provides start/pause/resume/abort control primitives.
- `apps/web/src/stores/executionStore.ts` already tracks execution lifecycle and checkpoint state.
- `apps/server/src/orchestrator/claude.ts` + `checkpoint.ts` already handle pause/checkpoint behavior and pending state semantics.
- `apps/server/src/api/routes/projects.ts` already returns project status and is the correct place to enrich paused-job visibility.
- Plan/Verify pages already exist and can host compact status strips without route-level redesign.

### Gaps this phase must close
- No unified cross-page orchestration control bar with a single source of truth.
- No explicit dependency/parallelism check endpoint contract for “can start another run?” decisions.
- No workflow dependency diagram that combines assessed/non-assessed/blocked/runnable state.
- No explicit resume-after-edit full revalidation gate in orchestration UX.

## Recommended Architecture

### 1) Project orchestration control plane
- Add a dedicated `orchestrationStore` for run registry, selected run, and per-project control state.
- Add a shared `OrchestrationControlBar` mounted in project-level layout.
- Add per-page `RunStatusStrip` components that read from the same store.

### 2) Dependency/parallelism contract
- Add server API route for parallelism assessment (e.g., per-phase run candidate):
  - Input: project, phase, candidate plan set.
  - Output: `allowed`, blocking dependencies/conflicts, assessed graph metadata.
- Enforce this contract before enabling Start for additional runs in same project.

### 3) Workflow diagram model
- Reuse existing plan metadata (`wave`, `depends_on`) and enrich with runtime states:
  - `assessed`, `runnable`, `blocked`, `active`, `paused`, `complete`.
- Render diagram on plan/execute surfaces with clear sequential edges and parallel groups.

### 4) Safety + revalidation path
- All high-impact actions go through centralized confirmation modal policy.
- On pause-edit-resume flow:
  - Lock editing while active.
  - On resume, run full revalidation before dispatch.
  - If revalidation fails, keep run paused and expose expandable failure reasons.

## Risks and Mitigations

- **Risk:** Parallelism checks diverge between UI and server.
  - **Mitigation:** Server is source of truth; UI only renders and gates on server response.
- **Risk:** Multi-run visibility becomes confusing.
  - **Mitigation:** Single global bar with selected-run context + per-page compact strip + dashboard summary.
- **Risk:** Resume-after-edit creates hidden behavior changes.
  - **Mitigation:** Revalidation summary explicitly lists changed tasks and impacted dependencies before resume.
- **Risk:** Confirmation fatigue from too many modals.
  - **Mitigation:** Consistent modal language + severity badges + optional detail expansion.

## Planning Implications

Best breakdown is 4 plans across 3 waves:
- Wave 1: control-plane foundation and server check contracts.
- Wave 2: parallelism diagram + lifecycle safety/revalidation UI.
- Wave 3: dashboard integration + end-to-end verification.

This split keeps architectural contracts first, allows independent UI tracks in parallel, and ends with integration hardening.
