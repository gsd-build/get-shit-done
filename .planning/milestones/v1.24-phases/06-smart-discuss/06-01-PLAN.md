<objective>
Replace the `Skill(skill="gsd:discuss-phase")` call in autonomous.md step 3a with inline smart discuss logic that proposes answers for grey areas in batch tables and lets the user accept or override them per area.

Purpose: The current autonomous workflow delegates to the interactive discuss-phase which asks open-ended questions one at a time. Smart discuss inverts this — the system does the analysis, proposes recommended answers in a table per grey area, and the user confirms or tweaks. This is faster for autonomous mode while preserving identical CONTEXT.md output for downstream agents.

**Structure the step with these clearly labeled sub-sections:**

**Sub-step 1: Load prior context**

**Sub-step 2: Scout codebase**

**Sub-step 3: Analyze phase and generate proposals**
Adapted from discuss-phase.md `analyze_phase` (lines 267-297) BUT extended to generate recommended answers. Instructions to:
- Get phase details: `DETAIL=$(node "$HOME/.claude/get-shit-done/bin/gsd-tools.cjs" roadmap get-phase ${PHASE_NUM})`
- Extract goal, requirements, success_criteria
- Determine domain type from goal:
  - Something users SEE → visual presentation, interactions, states
  - Something users CALL → interface contracts, responses, errors
  - Something users RUN → invocation, output, behavior modes
  - Something users READ → structure, tone, depth, flow
  - Something being ORGANIZED → criteria, grouping, handling exceptions
- Check prior decisions — skip grey areas already decided in prior phases
- Generate 3-4 grey areas with ~4 questions each
- **For each question, pre-select a recommended answer** based on: prior decisions (consistency), codebase patterns (reuse), domain conventions (standard approaches), ROADMAP success criteria
- Also generate 1-2 alternatives per question
- Annotate with prior decision context and code context where relevant

**Infrastructure detection (include in sub-step 3):**
Before generating grey areas, check if phase is pure infrastructure. A phase is infrastructure-only when:
1. Goal keywords match: "scaffolding", "plumbing", "setup", "configuration", "migration", "refactor", "rename", "restructure", "upgrade", "infrastructure"
2. AND success criteria are all technical: "file exists", "test passes", "config valid", "command runs"
3. AND no user-facing behavior is described (no "users can", "displays", "shows", "presents")

If infrastructure-only: skip proposal presentation, jump directly to write_smart_context with a minimal CONTEXT.md containing:
- `<domain>` with phase boundary from ROADMAP
- `<decisions>` with single "### Claude's Discretion" subsection noting "All implementation choices are at Claude's discretion — pure infrastructure phase"
- `<code_context>` with whatever the scout found
- `<specifics>` with "No specific requirements — infrastructure phase"
- `<deferred>` with "None"

Display: `Phase ${PHASE_NUM}: Infrastructure phase — skipping discuss, writing minimal context.`

**Sub-step 4: Present proposals per area (one at a time)**
For each grey area (M of N):

Display a table:
```
### Grey Area {M}/{N}: {Area Name}

| # | Question | ✅ Recommended | Alternative(s) |
|---|----------|---------------|-----------------|
| 1 | {question} | {answer} — {brief rationale} | {alt1}; {alt2} |
| 2 | {question} | {answer} — {brief rationale} | {alt1} |
| 3 | {question} | {answer} — {brief rationale} | {alt1}; {alt2} |
| 4 | {question} | {answer} — {brief rationale} | {alt1} |
```

Then use AskUserQuestion:
- header: "Area {M}/{N}"
- question: "Accept these answers for {Area Name}?"
- options: Build dynamically — always include "Accept all" first, then "Change Q1" through "Change QN" for each question (up to 4), then "Discuss deeper" last. Cap at 6 explicit options max (AskUserQuestion adds "Other" automatically).

**On "Accept all":** Record all recommended answers for this area. Move to next area.

**On "Change QN":** Use AskUserQuestion with the alternatives for that specific question:
- header: "{Area Name}"
- question: "Q{N}: {question text}"
- options: List the 1-2 alternatives plus "You decide" (maps to Claude's Discretion)
Record the user's choice. Re-display the updated table with the change highlighted (show ✅ on the new choice). Re-present the full acceptance prompt so the user can make additional changes or accept.

**On "Discuss deeper":** Switch to interactive mode for this area only — ask 4 questions one at a time using AskUserQuestion (like regular discuss-phase `discuss_areas` pattern, lines 373-436), with 2-3 concrete options per question plus "You decide". After 4 questions, ask "More questions about {area}, or move to next?" If "More", ask 4 more. If "Next", display final summary table of captured answers for this area and move on.

**On "Other" (free text):** Interpret as either a specific change request or general feedback. Incorporate into the area's decisions, re-display updated table, re-present acceptance prompt.
