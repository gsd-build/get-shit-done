## 1. System Classification

**System Type:** Code Automation Agent (with Multi-Agent decomposition)

**Description:**
The SME Creator Agent is a GSD agent (markdown agent definition) that analyzes a codebase's process-specific code paths, git history, PR descriptions, and documentation to produce structured Subject Matter Expert (SME) documents. These documents capture domain-specific risks, test gaps, outdated logic, edge cases, and blockers with severity labels (BLOCKER/WARNING/WATCH). The agent is used by developers building GSD-managed projects to pre-populate domain knowledge that the plan-phase gate later enforces. "Good" means the generated SME document accurately identifies real domain risks that a human expert would flag, captures the "why" behind patterns (not just the "what"), and doesn't miss critical edge cases or fragile logic.

**Critical Failure Modes:**
1. Missing critical domain risks — agent fails to identify fragile logic, edge cases, or known failure patterns that a human domain expert would immediately flag
2. False positives at BLOCKER severity — agent labels a non-issue as BLOCKER, causing the plan-phase gate to halt planning unnecessarily and eroding trust in the system
3. Shallow "why" analysis — agent describes what code does but misses the historical context (PRs, commit messages, code comments) that explains why patterns exist, producing SME documents that lack actionable insight
4. Context budget exhaustion — agent burns the session context window on a large codebase instead of using parallel sub-agent decomposition, producing incomplete analysis or crashing mid-run
5. Hallucinated findings — agent fabricates risks or code patterns that don't exist in the actual codebase, producing an SME document that misleads downstream consumers

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer Tooling / Software Engineering Knowledge Management

**User Population:** Software developers using GSD to manage project planning and execution. They invoke the SME Creator Agent once per business process (e.g., "contribution processing", "billing", "authentication") and then rely on the resulting SME document to catch domain risks automatically during every subsequent planning phase. They are technical practitioners, not domain novices — they will immediately recognize when a finding is shallow, fabricated, or misses an obvious known risk.

**Stakes Level:** High

**Output Consequence:** The SME document is consumed by the plan-phase gate (Phase 6) to block or warn on plans that miss domain risks. A missing BLOCKER finding causes a developer to write and execute a plan that breaks production behavior. A false BLOCKER finding halts planning unnecessarily and erodes trust until developers start bypassing the gate entirely, destroying the system's entire value proposition.

### What Domain Experts Evaluate Against

```
Dimension: Historical rationale capture
Good: Finding cites the specific commit hash, PR number, or code comment that explains
      WHY a pattern exists — e.g., "Member ID capped at 9 chars per PR #412 (HHS
      enrollment spec constraint, not business logic)."
Bad:  Finding describes what the code does — e.g., "Member ID is validated to 9
      characters" — without any reference to why, leaving the reader unable to assess
      whether the constraint is load-bearing.
Stakes: Critical
Source: Code Digital Twin research (arxiv.org/html/2510.16395v1) — tacit knowledge
        behind software (design rationales, historical decisions) decays over time and
        is seldom documented; agents that read only current code state reproduce exactly
        this decay rather than reversing it.
```

```
Dimension: Severity calibration
Good: BLOCKER is applied only when the finding describes a pattern that, if violated,
      would break a known production behavior or data contract — e.g., "Contribution
      amounts over $999.99 trigger a different processing path; any refactor that
      flattens this branch will silently drop high-value contributions (BLOCKER)."
Bad:  BLOCKER is applied to style concerns, theoretical edge cases with no evidence of
      real occurrence, or findings that duplicate existing linter rules — e.g.,
      "BLOCKER: this function has no JSDoc comment."
Stakes: Critical
Source: Production developer tooling practice — false positives at BLOCKER severity
        cause developers to bypass gates; Microsoft CORE framework showed 25.8% false
        positive reduction was the primary adoption driver (digitalapplied.com/blog/
        ai-code-review-automation-guide-2025).
```

```
Dimension: Finding traceability to actual code
Good: Every risk finding names the specific file path, function name, and line range
      where the fragile logic lives — e.g., "Fraud check bypass condition in
      `src/contributions/processor.ts:handleResubmit()` lines 84-97."
Bad:  Findings are stated as general process-level assertions with no pointer into
      the codebase — e.g., "The contribution processing flow has fraud risks."
Stakes: High
Source: REQUIREMENTS.md AUDIT-04 — auditor must require "concrete mitigations that
        name file paths and function calls, not abstract patterns"; same standard
        applies to the upstream creator output.
```

```
Dimension: Test gap specificity
Good: Test gap finding names the scenario that is untested, explains why it is risky,
      and references the code path that exercises it — e.g., "No test covers the
      zero-amount contribution edge case; `processContribution()` has a divide-by-amount
      operation at line 201 that will throw on zero input."
Bad:  Test gap finding states coverage percentage or lists files with low coverage
      without identifying which missing scenarios are actually dangerous — e.g.,
      "Coverage for contributions module is 71%."
Stakes: High
Source: Survey of Bugs in AI-Generated Code (arxiv.org/html/2512.05239v1) — edge case
        handling is the dominant bug category in LLM-analyzed code; generic coverage
        metrics do not identify which uncovered paths carry production risk.
```

```
Dimension: Outdated logic identification
Good: Outdated logic finding cites the original context that made the code correct
      (a regulation, a third-party API version, a business rule) and the evidence that
      context has since changed — e.g., "COVID-era contribution limit suspension logic
      in `adjustLimits()` references IRS Notice 2020-33; that notice expired 2021-12-31
      and the code now applies suspended limits unconditionally (WARNING)."
Bad:  Outdated logic finding flags code that looks old by superficial markers (old date
      in comment, deprecated API name) without evidence that the code's behavior is
      incorrect under current rules.
Stakes: High
Source: ROADMAP.md Phase 3 success criterion 4 — the HSA engine target explicitly
        requires COVID-era logic to be identified as a domain risk; the pattern
        generalizes to any time-bounded business rule.
```

### Known Failure Modes in This Domain

1. **Context-only analysis (missing git archaeology):** The agent reads current file state but does not invoke `git log --follow`, `git blame`, or PR descriptions. The resulting SME document describes current behavior accurately but contains zero historical rationale. A developer reading it cannot distinguish a load-bearing constraint from an arbitrary implementation choice. This is the dominant failure mode for code-analysis agents per the Code Digital Twin research — tacit knowledge lives in history, not in current state.

2. **Severity inflation from pattern-matching:** The agent applies BLOCKER to findings where the triggering signal is surface-level (a `TODO` comment, an absence of error handling, a deprecated function name) rather than evidence of a real production failure path. Developers encounter false BLOCKERs immediately on first use, lose trust in severity labels, and begin ignoring or bypassing the gate — eliminating the system's core value. Research on AI code review tools identifies false positives as the leading adoption blocker (65% of developers cite missing context as the primary failure driver).

3. **Fabricated findings with plausible code references:** The agent generates a finding that names a real file and function but describes behavior that does not exist in that location. Because the file and function name are real, the finding looks credible. Downstream consumers (the plan-phase gate, the developer reading the SME document) may spend significant time investigating a non-issue before discovering the fabrication. Nearly 20% of AI code analysis suggestions reference non-existent patterns per 2025 research on LLM code review hallucinations.

4. **Process boundary over-reach:** The agent is given a process name (e.g., "contribution processing") but crawls the entire codebase, returning findings about authentication, billing, and infrastructure that are not relevant to the specified process. The SME document becomes too large to be useful as a gate input and buries the actual process-specific risks. Research on agentic code systems identifies scope drift — agents that don't respect the boundaries of their task — as a distinct failure mode from hallucination.

### Regulatory / Compliance Context

No external regulatory requirements directly govern the SME Creator Agent itself — it is an internal developer tool with no user-facing output and no regulated data handling. However, the **business processes it analyzes** (contribution processing, benefits administration, healthcare billing) in the primary target domain (health savings accounts / benefits platforms) are subject to IRS regulations, HIPAA, and HHS enrollment specifications. The agent's output quality directly affects whether developers correctly identify compliance-relevant logic. A missed finding about an IRS contribution limit boundary or an HHS member ID constraint is not a compliance violation by the tool itself, but it increases the probability that a developer makes a planning mistake that becomes one.

The SME document format (SCHEMA-03: `last_analyzed_commit`) and staleness tracking (REFRESH-04) are design responses to this indirect compliance adjacency — stale domain knowledge is treated as a risk signal.

## 2. Framework Decision

**Selected Framework:** Claude Agent SDK (`@anthropic-ai/claude-agent-sdk`)

**Version:** ^0.2.84 (already in project dependencies)

**Rationale:**
Already the sole AI dependency in both `package.json` and `sdk/package.json`. The SME Creator Agent is a markdown agent definition spawned via this SDK, matching the exact pattern of gsd-planner, gsd-executor, and all other agents in the project. Built-in filesystem and shell access directly supports `git log --follow` and PR description analysis (CREATE-04), and parallel sub-agent spawning covers the CREATE-02 decomposition pattern without any additional dependencies.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangGraph (TypeScript) | Adds graph overhead not justified for this use case. Would be relevant if long-running SME creation across very large codebases required node-level checkpointing, but current architecture handles this via parallel sub-agents. |
| CrewAI | Python only — project is TypeScript. |
| LlamaIndex | RAG-focused — this is code analysis, not document retrieval. |
| OpenAI Agents SDK | Project is committed to Anthropic/Claude models. |

**Vendor Lock-In Accepted:** Yes — project has already consciously chosen Anthropic/Claude as the model provider. All existing agents use the same SDK.

### Entry Point Pattern — Code Automation Agent with Sub-Agent Decomposition

The `gsd-sme-creator` agent is a **markdown agent definition** (`agents/gsd-sme-creator.md`),
not TypeScript code. The SDK spawns it when the `create-sme.md` workflow calls `Task()`.

The TypeScript entry point (in the GSD SDK) wraps `query()` like this:

```typescript
import { query } from '@anthropic-ai/claude-agent-sdk';
import { readFileSync } from 'node:fs';

// Load the agent definition markdown
const agentDef = readFileSync('./agents/gsd-sme-creator.md', 'utf-8');

// How the GSD SDK launches gsd-sme-creator (mirrors runPhaseStepSession in session-runner.ts)
const queryStream = query({
  prompt: `Create SME document for process: ${processName}`,
  options: {
    // Use Claude Code's built-in system prompt (gives Bash, Read, Write, Grep, Glob)
    systemPrompt: {
      type: 'preset',
      preset: 'claude_code',
      // Agent's markdown definition is appended after the preset
      append: agentDef,
    },
    // REQUIRED: loads CLAUDE.md project conventions — omitting this is a common mistake
    settingSources: ['project'],
    // Task must be listed here to enable sub-agent spawning
    allowedTools: ['Read', 'Bash', 'Grep', 'Glob', 'Write', 'Task'],
    permissionMode: 'bypassPermissions',
    allowDangerouslySkipPermissions: true,
    maxTurns: 80,
    maxBudgetUsd: 10.0,
    model: 'claude-sonnet-4-6',
    cwd: process.cwd(),
  },
});

// Iterate the stream — result message is always last
for await (const message of queryStream) {
  if (message.type === 'result') {
    const success = message.subtype === 'success';
    console.log(`SME creation ${success ? 'complete' : 'failed'}. Cost: $${message.total_cost_usd}`);
  }
}
```

The agent markdown itself (`agents/gsd-sme-creator.md`) uses `Task()` calls to spawn
sub-agents. This is pure markdown — no TypeScript required in the agent file:

```markdown
---
name: gsd-sme-creator
description: Analyzes a codebase process and writes a complete SME document.
tools: Read, Bash, Grep, Glob, Write, Task
color: "#F59E0B"
---

<role>
You are a GSD SME creator. Given a PROCESS_NAME, analyze its code paths,
git history, PR descriptions, and docs. Write the complete
.planning/smes/{PROCESS_NAME}-SME.md.
</role>

<process>
<step name="spawn_analyzers" condition="Task tool available">
Task(
  subagent_type="gsd-sme-creator-analyzer",
  description="Analyze {process} code paths",
  run_in_background=true,
  prompt="Process: {PROCESS_NAME}
Files: {file_list}
Output: .planning/smes/.tmp/{PROCESS_NAME}-part-{N}.md
Run git log --follow on each file. Extract risks and why."
)
</step>
</process>
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| **Agent definition (`.md` file)** | Markdown file with YAML frontmatter (`name:`, `tools:`, `color:`) and a `<role>` block defining the agent's instructions. The SDK uses `name:` as the `subagent_type` identifier. | Every agent in this project. `gsd-sme-creator.md` is the file you create in Phase 3. The `name:` field must exactly match the `subagent_type` string in any `Task()` call that spawns it. |
| **`query()` function** | The sole TypeScript SDK entry point. Spawns a Claude session with a prompt, system prompt configuration, tool permissions, and optional programmatic subagent definitions. Returns `AsyncIterable<SDKMessage>`. | Used in `sdk/src/session-runner.ts` to run every plan and phase step. The SME creator workflow will invoke it via `runPhaseStepSession()`. |
| **`Task` tool** | Built-in tool (must appear in both `allowedTools` in `query()` options AND in the agent's YAML `tools:` frontmatter) that lets an orchestrator agent spawn a named sub-agent with a fresh context window. `subagent_type` must match the `name:` of an agent definition file. | The SME creator's parallel decomposition step — spawns analyzer sub-agents for large code path partitions. Mirrors the pattern in `get-shit-done/workflows/map-codebase.md`. |
| **`systemPrompt` preset + append** | `{ type: 'preset', preset: 'claude_code', append: <agent-def-content> }` — gives the session Claude Code's full system prompt (filesystem and shell tool capabilities) and appends the agent's own instructions on top. | Required for any agent that uses `Read`, `Bash`, `Grep`, `Glob`, or `Write`. The `claude_code` preset alone does NOT load `CLAUDE.md`; add `settingSources: ['project']` to get project conventions. |
| **`settingSources: ['project']`** | Tells the SDK to load `CLAUDE.md` from the working directory into the session context. Without this, all project conventions, naming rules, and security constraints from `CLAUDE.md` are invisible to the agent. | Every invocation that runs in a project context. Already set in `sdk/src/session-runner.ts`. Must be explicitly set — it does not default on. |

### Common Pitfalls

1. **Omitting `Task` from `allowedTools` silently breaks sub-agent spawning.** The agent receives no error — it simply cannot call `Task`, so the parallel decomposition pattern falls back to nothing (no sequential fallback is automatic). Always verify `'Task'` appears in `allowedTools` in the `query()` call AND in the agent's YAML `tools:` frontmatter. Both are required; either alone is insufficient.

2. **`subagent_type` mismatch with agent definition `name:` field.** The `subagent_type` string in a `Task()` call must exactly match the `name:` field in the target agent's YAML frontmatter. A mismatch causes the SDK to fall back to a generic unspecialized agent with no role instructions. There is no warning or error — the sub-agent just runs without your prompt. See `map-codebase.md`'s `<available_agent_types>` block for the correct pattern.

3. **Omitting `settingSources: ['project']` makes `CLAUDE.md` invisible to the agent.** The `claude_code` preset does NOT automatically load `CLAUDE.md`. This is documented in the SDK migration guide as a breaking change from v0.0.x. Without it, the agent lacks all project conventions — TypeScript naming rules, file structure, security restrictions — and will produce code and documents that violate them.

4. **`git log` without `--follow` truncates history at file renames.** For CREATE-04, the agent must use `git log --follow <file>` to trace files across renames and moves. Without `--follow`, history stops at the most recent rename, which in TypeScript projects happens frequently during module path refactoring. The "why" behind patterns — the primary value of the SME document — is precisely the history that renames would hide.

5. **Reading sub-agent output files before `TaskOutput(block: true)` completes.** After spawning sub-agents with `run_in_background=true`, the orchestrator must wait for each agent to finish via `TaskOutput` with `block: true` before reading their output files. Reading immediately after spawning produces empty or partial content. See the `collect_confirmations` step in `get-shit-done/workflows/map-codebase.md` for the correct await pattern.

## 4. Implementation Guidance

**Model Configuration:**

```typescript
// Primary model — matches project's "balanced" profile in session-runner.ts
model: 'claude-sonnet-4-6'

// Token budget — SME creation for a large process is expensive in turns, not output tokens
// The agent writes via the Write tool, so response text is short (confirmation only)
maxTurns: 80           // orchestrator needs turns for: partition, spawn, wait, collect, synthesize, write
maxBudgetUsd: 10.0     // upper guard; typical mid-sized process run costs $0.50-1.50

// No explicit max_tokens on response output — the SME document is written via Write tool,
// not returned as response text. Response text is only used for completion confirmation.

// Sub-agent analyzers use the same model (haiku is appropriate only for v2 classification sub-tasks)
// subagent_model: 'claude-sonnet-4-6'  // implicitly inherited from session
```

**Core Pattern:**

The SME creator follows the exact same parallel decomposition pattern as `gsd-map-codebase`. There are two agents and two roles:

**Orchestrator** (`gsd-sme-creator.md`):
1. Detect `Task` tool availability
2. Identify all files belonging to the named process (Grep + Glob)
3. Partition files into groups of 5-10 files
4. Spawn parallel analyzer sub-agents with `run_in_background=true`
5. Wait for all sub-agents via `TaskOutput(block: true)`
6. Read partial findings from each temp file
7. Merge, deduplicate, apply severity labels
8. Write final `.planning/smes/{PROCESS_NAME}-SME.md`
9. Clean up `.planning/smes/.tmp/`

**Analyzer sub-agent** (`gsd-sme-creator-analyzer.md`):
1. Receive file list and output path
2. Read each file for code pattern analysis
3. Run `git log --follow -n 50 <file>` for history
4. Run `git log --grep="Merge pull request" -n 10 <file>` for PR descriptions
5. Identify risks, test gaps, outdated logic, edge cases
6. Write compressed findings to designated temp file
7. Return brief confirmation (finding count only — not full content)

The key insight from `map-codebase.md`: **sub-agents write files, orchestrator accumulates only summaries.** The orchestrator never holds raw code or full git history in its context.

```markdown
<!-- Skeleton: agents/gsd-sme-creator.md -->
---
name: gsd-sme-creator
description: Analyzes a codebase process and produces a structured SME document.
tools: Read, Bash, Grep, Glob, Write, Task
color: "#F59E0B"
---

<role>
You are a GSD SME creator. Given a PROCESS_NAME, you analyze its code paths,
git history, PR descriptions, and docs to produce .planning/smes/{PROCESS_NAME}-SME.md.
Sub-agents write directly to .tmp files. You synthesize their findings into the final doc.
</role>

<process>

<step name="detect_capabilities">
Check if the Task tool is available. If not, perform analysis sequentially in-context
(4 passes: code patterns, git history, test gaps, synthesis).
</step>

<step name="partition_code_paths">
Identify files belonging to this process:
```bash
grep -rl "{PROCESS_NAME}" --include="*.ts" sdk/src/ agents/ get-shit-done/
find . -name "*{process_name}*" -not -path "*/node_modules/*"
```
Group into partitions of 5-10 files. Create .planning/smes/.tmp/ directory.
</step>

<step name="spawn_analyzers" condition="Task tool available">
For each partition N, spawn with run_in_background=true:

Task(
  subagent_type="gsd-sme-creator-analyzer",
  description="Analyze {process} files partition {N}",
  run_in_background=true,
  prompt="Process: {PROCESS_NAME}
Files: {comma_separated_file_list}
Output: .planning/smes/.tmp/{PROCESS_NAME}-part-{N}.md
Analyze each file for risks, test gaps, outdated logic, edge cases.
Run: git log --follow -n 50 <file> for each file.
Capture WHY patterns exist (PR numbers, commit hashes, code comments).
Write findings in BLOCKER/WARNING/WATCH format. Return finding count only."
)
</step>

<step name="collect_results">
Wait for all analyzers — call TaskOutput for each with block=true, timeout=300000.
Read each .planning/smes/.tmp/{PROCESS_NAME}-part-{N}.md.
</step>

<step name="synthesize_and_write">
Merge findings. Deduplicate (same file + same line range = same finding).
Apply BLOCKER only when evidence of real production failure path exists.
Write .planning/smes/{PROCESS_NAME}-SME.md with frontmatter including
last_analyzed_commit and block_mode.
Run self-check: grep -c required section headers (must be >= 4).
Delete .planning/smes/.tmp/ after successful write.
</step>

</process>
```

**Tool Use:**

| Tool | Role in SME creation | Required in |
|------|---------------------|-------------|
| `Read` | Load TypeScript source files for pattern analysis | Both orchestrator and analyzer |
| `Bash` | Run `git log --follow`, `git show`, `git log --grep` for history and PR data | Both |
| `Grep` | Find all files touching a process; locate `TODO`/`FIXME`/`HACK` comments | Both |
| `Glob` | Enumerate test files, spec files, docs for a process | Both |
| `Write` | Write partial findings (analyzer) and final SME document (orchestrator) | Both |
| `Task` | Spawn parallel analyzer sub-agents for large code path partitions (CREATE-02) | Orchestrator only |

Sub-agents (`gsd-sme-creator-analyzer`) do NOT need `Task` — they do not spawn further sub-agents.

**State Management:**

This project uses the filesystem as its sole state store — no database, no in-memory state across sessions.

- **Partial findings** are written by sub-agents to `.planning/smes/.tmp/{PROCESS_NAME}-part-{N}.md`
- **Final output** is `.planning/smes/{PROCESS_NAME}-SME.md` with YAML frontmatter:
  ```yaml
  ---
  process_name: contribution-processing
  last_analyzed_commit: abc1234def5678...   # git rev-parse HEAD at creation time
  block_mode: soft                           # configurable per SCHEMA-04
  generated_at: 2026-04-29T12:00:00Z
  finding_counts:
    blocker: 2
    warning: 4
    watch: 3
  ---
  ```
- **Staleness tracking** is via `last_analyzed_commit` — the plan-phase gate compares this against current `HEAD` (REFRESH-04)
- **No resumption** — if the orchestrator crashes mid-run, a fresh run overwrites any partial `.tmp` files
- **Atomic final write** — the orchestrator collects ALL findings first, then writes the final SME file in one `Write` tool call. Partial writes to the final output path risk a malformed document being consumed by the gate

**Context Window Strategy:**

The primary risk is Critical Failure Mode 4: context budget exhaustion. Mitigations:

1. **Parallel decomposition (CREATE-02):** For any process touching more than ~10 files, partition by file group and spawn parallel sub-agents. Each sub-agent gets a fresh 200k-token context window. The orchestrator only accumulates sub-agent result summaries (finding counts + compressed findings), not raw code.

2. **Sub-agent write-directly pattern (from `map-codebase.md`):** Sub-agents write partial findings to temp files and return only a brief confirmation to the orchestrator. The orchestrator never accumulates raw code in its context — only compressed finding text.

3. **Git log depth limiting:** Use `git log --follow -n 50 <file>` — 50 commits captures ~6-12 months of typical development without unbounded token consumption. For PR descriptions, `git log --grep="Merge pull request" --format="%s%b" -n 20` is sufficient.

4. **Sequential fallback (when `Task` unavailable):** Perform analysis in 4 in-context passes, writing after each pass to stay below 50% context per pass:
   - Pass 1: Code pattern analysis → write `## Identified Risks` draft
   - Pass 2: Git history → annotate findings with "why" evidence
   - Pass 3: Test gap analysis → write `## Test Gaps` section
   - Pass 4: Synthesis → write final SME document

5. **SDK automatic compaction:** The Claude Agent SDK compacts context at ~80% usage automatically. Write partial findings before reaching 60% context (safety margin). Compaction is safe for file-analysis tasks — the agent can continue after compaction without losing tool access.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### 4b.1 Structured Outputs

**Note on tooling:** This project is TypeScript + Claude Agent SDK, not Python. "Pydantic" maps to **Zod schemas** for TypeScript runtime validation, or to the SDK's `outputFormat` option for JSON schema enforcement at the API level.

**SME document validation schema (Zod):**

```typescript
import { z } from 'zod'; // npm install zod (add to sdk/package.json if not present)

// Severity label enum — matches SCHEMA-02
const SeverityLabel = z.enum(['BLOCKER', 'WARNING', 'WATCH']);

// Individual finding — enforces the "why" requirement
const SMEFinding = z.object({
  severity: SeverityLabel,
  category: z.enum(['risk', 'test_gap', 'outdated_logic', 'edge_case', 'blocker']),
  title: z.string().min(10).max(120),
  description: z.string().min(20),
  file_path: z.string().optional(),    // must reference a real file (agent verifies with Read)
  why: z.string().min(20),             // required — historical rationale (PR, commit, comment)
  evidence: z.string(),                // git commit hash, PR number, or exact code comment
});

// SME document frontmatter — matches SCHEMA-01 through SCHEMA-04
const SMEDocumentFrontmatter = z.object({
  process_name: z.string(),
  last_analyzed_commit: z.string().regex(/^[0-9a-f]{7,40}$/),  // git SHA (short or full)
  block_mode: z.enum(['soft', 'strict']),
  generated_at: z.string().datetime(),
  finding_counts: z.object({
    blocker: z.number().int().min(0),
    warning: z.number().int().min(0),
    watch: z.number().int().min(0),
  }),
});

type SMEFinding = z.infer<typeof SMEFinding>;
type SMEDocumentFrontmatter = z.infer<typeof SMEDocumentFrontmatter>;
```

**How the framework integrates:**

The Claude Agent SDK provides `outputFormat` for JSON schema enforcement on the agent's final result message. Use this in `create-sme.md` workflow to validate the agent returned a completion summary before writing to disk:

```typescript
// In the workflow's query() invocation (not the agent markdown)
const queryStream = query({
  prompt: `Create SME document for process: ${processName}`,
  options: {
    // ... other options ...
    // Enforce that the agent's result message matches this schema
    outputFormat: {
      type: 'json_schema',
      schema: {
        type: 'object',
        properties: {
          process_name: { type: 'string' },
          sme_path: { type: 'string', description: 'Absolute path to written SME file' },
          finding_counts: {
            type: 'object',
            properties: {
              blocker: { type: 'integer', minimum: 0 },
              warning: { type: 'integer', minimum: 0 },
              watch: { type: 'integer', minimum: 0 },
            },
            required: ['blocker', 'warning', 'watch'],
          },
          completed: { type: 'boolean' },
        },
        required: ['process_name', 'sme_path', 'finding_counts', 'completed'],
      },
    },
  },
});
```

**Retry logic:**

Structure validation happens at two levels:

1. **Agent self-check (in agent markdown, before exiting):**
   ```bash
   # Verify all required sections exist in the written SME document
   SECTION_COUNT=$(grep -c "## Process Overview\|## Identified Risks\|## Test Gaps\|## Edge Cases" \
     .planning/smes/${PROCESS_NAME}-SME.md)
   ```
   If count is less than 4, the agent rewrites missing sections. Maximum 2 self-repair attempts. On second failure, write the document with a `## INCOMPLETE` marker and exit with the section count in the result.

2. **Workflow-level validation (in `create-sme.md`):** After agent completes, the workflow:
   - Verifies `last_analyzed_commit` is present in frontmatter (required for staleness tracking)
   - Verifies at least one finding exists (a zero-finding SME document is likely a failure, not a clean codebase)
   - On failure: logs session ID and raw output, surfaces a warning to the user with path to the malformed file — does not retry automatically (SME creation is expensive; retry should be user-initiated)

### 4b.2 Async-First Design

**How async works in this framework:**

The Claude Agent SDK's `query()` returns an `AsyncIterable<SDKMessage>`. The only correct consumption pattern is `for await`. The GSD SDK wraps this in `processQueryStream()` in `sdk/src/session-runner.ts` — new code should follow that pattern exactly, not reimplement iteration.

**The one common mistake — synchronous child processes inside async stream processing:**

```typescript
// WRONG: execSync blocks the Node.js event loop
// This stalls the query() stream processing loop while git runs
import { execSync } from 'node:child_process';
// Do NOT call execSync() while iterating a query() stream
const history = execSync('git log --follow -n 50 file.ts').toString();

// CORRECT: git commands belong inside the agent's Bash tool calls
// The SDK runs Bash as a subprocess — this is already async from the SDK's perspective
// The TypeScript layer should only consume the message stream and extract the result
```

**Sub-agent parallelism is SDK-level, not JavaScript-level:**

Parallel sub-agents (spawned via `Task` with `run_in_background=true`) run as separate Claude Code processes managed by the SDK. This is not `Promise.all()`. Do not attempt to run multiple `query()` calls in parallel expecting them to share tool coordination — each `query()` is an independent session. The orchestrator's single `query()` session manages all sub-agents internally via the `Task`/`TaskOutput` tool pair.

**Stream vs. await for SME creation:**

- **`for await` (not UI streaming):** SME creation is a CLI background operation with no real-time display requirement. Use `for await` and extract the final `result` message. `processQueryStream()` in `session-runner.ts` already implements this correctly.
- **Streaming to UI:** Only relevant if SME creation is exposed via a web interface in a future phase. For CLI, awaiting the result message is correct.

### 4b.3 Prompt Engineering Discipline

**System vs. user prompt separation:**

In the Claude Agent SDK:
- **System prompt** (`systemPrompt.preset + append`): The agent's identity, tools list, process steps, output format rules, severity definitions, and inline few-shot examples. The full content of `agents/gsd-sme-creator.md` is the `append`. Static per agent version.
- **User prompt** (`prompt` parameter): The specific invocation — process name, target files, `today's date`, any runtime flags. Changes per invocation.

Never embed per-invocation data (process name, file list, dates) into the agent markdown file. The agent definition is a template; the workflow passes runtime values through the `prompt` parameter.

**Few-shot guidance for severity calibration:**

Include 2-3 inline examples in the agent's `<role>` block — one BLOCKER, one WARNING, one WATCH. This calibrates severity judgment more reliably than written rules alone. Inline is correct here (not dynamic retrieval) because the severity structure is fixed and does not depend on the codebase being analyzed.

```markdown
<!-- In agents/gsd-sme-creator.md — include in <role> or <severity_examples> block -->

BLOCKER example:
> **BLOCKER: phase-runner.ts allows concurrent phase execution**
> File: `sdk/src/phase-runner.ts`, function `run()`, lines 91-200
> Why: Concurrent phase guard was removed in commit a4b2c1d (PR #47) to unblock a
> CLI timeout issue. The state machine was never made thread-safe. Two concurrent
> `/gsd-execute-phase` calls write to STATE.md simultaneously, corrupting phase state.
> Evidence: `git log --grep="concurrent" sdk/src/phase-runner.ts` — see PR #47 comment
> "deferred the locking fix to post-GA"

WARNING example:
> **WARNING: sme.detect-processes uses keyword matching, not semantic analysis**
> File: `sdk/src/query/sme-detect-processes.ts`
> Why: Introduced in phase-2 as an MVP. Processes named with abbreviations (e.g.
> "contrib" vs "contribution") will not match. Phase touching "PR workflows" won't
> detect "pull-request" SME. Known limitation per code comment line 47.
> Evidence: `// TODO: semantic matching — keyword MVP only (phase-2)`
```

**Token budget:**

The SME creator agent should not have `max_tokens` on its response because it writes via the `Write` tool. Response text is only the completion confirmation (short) or the JSON result if `outputFormat` is set (bounded by schema).

Set `maxTurns: 80` — not unbounded. Each `git log` invocation, file read, and write costs a turn. 80 turns is generous for a large process; most complete in 30-50 turns.

### 4b.4 Context Window Management

**The core strategy: agents write files, orchestrator accumulates only summaries.**

This is the same pattern proven by `gsd-map-codebase` and `gsd-codebase-mapper`. The orchestrator never holds raw code or full git history in its context window. Sub-agents do the heavy reading and write findings to temp files. The orchestrator reads only compressed summaries (finding count + severity breakdown + top-level finding text).

**Partition sizing for sub-agents:**

Target 5-10 files per sub-agent partition. Signals a partition is too large:
- More than 15 files in the process's primary code path
- A single file exceeds 500 lines (e.g., a 1200-line monolith warrants its own sub-agent)
- The process spans multiple subsystems (SDK layer + workflow layer + agent layer each get their own partition)

At 5-10 files per partition with `git log -n 50` per file, each sub-agent uses approximately 20-35% of its 200k-token context window — well within the safe zone.

**Git history depth:**

`git log --follow -n 50` per file captures approximately 6-12 months of typical development without unbounded token consumption. For PR descriptions, `git log --grep="Merge pull request" --format="%s%n%b" -n 20` is sufficient. Avoid `git log --all` — it includes all branches and can be orders of magnitude larger.

**SDK automatic compaction:**

The Claude Agent SDK automatically compacts context at ~80% usage. For the SME creator:
- Write partial findings to temp files before reaching 60% context (safety margin before compaction)
- The safe checkpoint is after each file analysis pass, not mid-file
- Compaction is safe for file-analysis tasks — tool access and instructions survive compaction
- Do not structure analysis to depend on text written earlier in the conversation being literally present — it may be compacted

**Sequential fallback context budget (when `Task` unavailable):**

Four in-context passes, each targeting <50% context:
- Pass 1 (code pattern analysis): read source files, identify structural risks, write to `## Identified Risks` draft section
- Pass 2 (git history): `git log --follow` per file, annotate findings with "why" evidence
- Pass 3 (test gap analysis): read test files, identify untested dangerous scenarios
- Pass 4 (synthesis): merge and write final `.planning/smes/{PROCESS_NAME}-SME.md`

### 4b.5 Cost and Latency Budget

**Per-run cost estimate at `claude-sonnet-4-6` pricing (~$3/MTok input, ~$15/MTok output):**

| Process size | Files analyzed | Estimated turns | Estimated cost |
|---|---|---|---|
| Small (feature flag, config) | 5-10 files, <500 total lines | 20-30 turns | ~$0.15-0.40 |
| Medium (typical GSD process) | 10-25 files, ~2000 lines + git | 35-55 turns | ~$0.50-1.50 |
| Large (phase runner, SDK core) | 25-50 files, with full git history | 60-80 turns | ~$1.50-4.00 |

`maxBudgetUsd: 10.0` in session configuration guards against runaway analysis on unexpectedly large processes. The typical run for a GSD-managed project process costs $0.75-1.50.

**Prompt caching:**

The Claude Agent SDK automatically caches prompt prefixes (visible in `cache_read_input_tokens` in `session-runner.ts`'s `extractUsage()`). For the SME creator:
- The agent's system prompt (full markdown definition) is cached after the first turn — cache hit rate is high for subsequent turns in the same session
- `CLAUDE.md` loaded via `settingSources: ['project']` is also cached
- File content re-read within a session (e.g., reading `phase-runner.ts` for code patterns then again for test gap analysis) benefits from caching — no special action needed

**No semantic caching for SME creation:** Each run targets a specific process and commit state. Stale cached analysis would produce outdated findings that defeat `last_analyzed_commit` staleness tracking (SCHEMA-03, REFRESH-04). Do not implement semantic caching for this use case.

**Cheaper models for sub-tasks (v2 only):**

Use `claude-sonnet-4-6` for all agents in v1. In v2, `claude-haiku-4-5` is appropriate for:
- **Severity classification:** Given a finding description, route to BLOCKER/WARNING/WATCH. This is a short-context classification task with fixed input/output.
- **Duplicate detection:** Given two findings, determine if they describe the same risk.
- **Section routing:** Given a code block, determine which SME section (risk/test_gap/outdated_logic/edge_case) it belongs to.

Migrate to Haiku for these sub-tasks only after v1 produces calibrated examples to verify Haiku accuracy — do not downgrade model quality before establishing a baseline.

## 5. Evaluation Strategy

### Dimensions

The five dimensions below map directly to the domain rubric ingredients in Section 1b. Each dimension targets one of the critical failure modes and is grounded in specific domain expert criteria, not generic labels.

| Dimension | Pass | Fail | Measurement | Priority |
|-----------|------|------|-------------|----------|
| **Historical rationale capture** | Every BLOCKER or WARNING finding cites a specific commit hash, PR number, or code comment explaining WHY the constraint exists — e.g., "Member ID capped per PR #412 (HHS spec)" | Finding describes what code does without any reference to why — e.g., "Member ID validated to 9 characters" with no historical reference | LLM Judge (calibrated against domain expert labels from reference dataset) | Critical |
| **Severity calibration** | BLOCKER is applied only when the finding names a pattern whose violation would break a known production behavior or data contract; WARNING and WATCH are used for lower-stakes findings | BLOCKER applied to style issues, theoretical edge cases with no production evidence, or findings that duplicate existing linter rules — e.g., "BLOCKER: this function has no JSDoc comment" | LLM Judge + Human (human review of all BLOCKER labels in reference dataset; LLM judge scores non-BLOCKER runs) | Critical |
| **Finding traceability to actual code** | Every risk finding names a specific file path, function name, and line range where the fragile logic exists; file path is verified to exist in the repository | Finding states a process-level assertion with no pointer into the codebase — e.g., "The contribution processing flow has fraud risks" — or names a file/function that does not exist | Code (file existence check via `ls`/`stat`; regex to verify `file:function():lines` pattern present in each finding) | High |
| **Test gap specificity** | Test gap finding names the untested scenario, explains why it is dangerous, and references the code path that exercises it | Test gap finding reports a coverage percentage or lists files without identifying which uncovered scenarios carry production risk | LLM Judge (calibrated against developer-labeled reference examples distinguishing actionable vs. noise test gap findings) | High |
| **Outdated logic identification** | Outdated logic finding cites the original context (regulation, API version, business rule) that made the code correct AND presents evidence the context has changed — e.g., cites IRS Notice 2020-33 expiration date | Outdated logic finding flags code based on superficial age markers (old date in comment, deprecated name) without evidence the behavior is now incorrect | LLM Judge + Human (domain expert validates whether cited context change is real; LLM judge scores presence of evidence citation) | High |
| **Process boundary adherence** | All findings reference files and functions within the specified process's code paths; no findings about unrelated processes (authentication, billing) appear in a "contribution-processing" SME | SME document for process X contains multiple findings about process Y's code, burying the relevant findings | Code (grep cited file paths against the process's known file set; flag findings where >30% of files are outside the process boundary) | High |
| **Schema and frontmatter completeness** | Output file passes Zod schema validation: all required frontmatter fields present (`process_name`, `last_analyzed_commit` as valid git SHA, `block_mode`, `generated_at`, `finding_counts`); all required section headers present (Process Overview, Identified Risks, Test Gaps, Edge Cases) | Frontmatter missing `last_analyzed_commit`, `finding_counts` keys absent or wrong type, or fewer than 4 required section headers in the document body | Code (Zod schema parse; `grep -c` section header check; git SHA regex validation) | Critical |
| **Task completion** | Agent produces a non-empty `.planning/smes/{PROCESS_NAME}-SME.md`, exits with `completed: true` in the result message, finding_counts sum > 0, and `.tmp/` directory cleaned up | Agent exits with error subtype, writes zero findings, leaves `.tmp/` directory populated, or produces an `## INCOMPLETE`-marked document | Code (file existence check, result message subtype, finding_counts sum, tmp directory cleanup check) | Critical |

**Rubric calibration note:** LLM judge dimensions (historical rationale, severity calibration, test gap specificity, outdated logic) require calibration before being trusted in CI/CD. Calibration target: ≥ 0.75 agreement with domain expert labels on the reference dataset. Run calibration using the reference dataset before enabling LLM judge checks in the CI/CD gate.

## 6. Guardrails

### Online (Real-Time)

This system is an internal developer tool, not a user-facing consumer product. The output is a markdown file consumed by a downstream agent, not a real-time response to an end user. Online guardrails are therefore minimal — the two catastrophic failure modes that warrant real-time intervention are fabricated file references (which waste developer investigation time immediately) and budget exhaustion (which wastes money and leaves no output).

| Guardrail | Trigger Condition | Intervention | Implementation |
|-----------|------------------|--------------|----------------|
| **Fabricated file path detection** | Any finding in the output SME document cites a file path that does not exist in the repository (`stat()` fails) | Block the SME document from being written to `.planning/smes/`; surface a structured error to the workflow listing the fabricated paths; do not write a partial document | Workflow-level check in `create-sme.md` after agent completes: `for each cited .ts path in output: assert fs.existsSync(path)`. Runs before the final Write step. |
| **Budget cap enforcement** | Session `total_cost_usd` in the SDK result message exceeds `maxBudgetUsd: 10.0` (SDK enforces this automatically) | SDK terminates the session; workflow logs the partial output path and cost; user is notified to re-run with a narrower process scope or larger budget | Built into `query()` options — `maxBudgetUsd: 10.0` is already specified. Workflow catches `SDKResultError` with `subtype: 'budget_exceeded'` and surfaces a user-readable message. |
| **Zero-finding output** | Agent completes successfully (`completed: true`) but `finding_counts.blocker + finding_counts.warning + finding_counts.watch === 0` | Flag as likely agent failure (not a clean codebase); do not silently accept a zero-finding SME as valid; surface a warning requiring user acknowledgment before the document is used by the gate | Workflow-level check: if `findingCounts.total === 0`, write document with a `## WARNING: ZERO FINDINGS` banner and require explicit user confirmation before `create-sme.md` marks the workflow complete |

### Offline (Flywheel)

Offline metrics feed the improvement loop for model prompt tuning and agent instruction updates. These run on a sampled batch after production use, not on every invocation.

| Metric | What It Measures | Sampling Strategy | Action on Degradation |
|--------|-----------------|------------------|-----------------------|
| **BLOCKER precision** | Fraction of BLOCKER-labeled findings that a domain expert agrees are genuine production-breaking risks (not false positives) | Sample all runs where `finding_counts.blocker > 0`; domain expert reviews BLOCKER findings monthly or after any agent prompt update | If BLOCKER precision drops below 70% on any 5-run sample, pause BLOCKER usage in strict-mode gate until prompt is recalibrated; route all gate decisions through soft-mode temporarily |
| **Historical rationale coverage rate** | Fraction of findings that include a commit hash, PR number, or code comment as evidence (measurable by regex) | 20% random sample of all completed SME runs; automated regex check (`/commit [0-9a-f]{7,}|PR #\d+|\/\/ /`) | If coverage rate drops below 80%, investigate whether `git log --follow` is being invoked (check session traces in Phoenix); update analyzer prompt to reinforce history capture |
| **Git archaeology invocation rate** | Whether the agent called `git log --follow` at least once per analyzed file (observable in Phoenix session traces via Bash tool call inspection) | 100% of runs during the first month post-deployment; 25% random sample thereafter | If invocation rate drops below 90% (e.g., agent skips git on small files), add an explicit `<requirement>` block to the analyzer agent definition |
| **End-to-end cost per run** | `total_cost_usd` from SDK result message, tracked per process size bucket (small/medium/large by file count) | 100% of runs — cost is always in the SDK result; no sampling needed | Alert if median cost for medium-sized processes exceeds $2.50 (1.7x the expected $1.50 ceiling); investigate whether partition sizing is working correctly or model is looping |
| **Developer bypass rate** | Rate at which developers invoke `--acknowledge-sme-risk` to override a BLOCKER finding in the plan-phase gate (Phase 6 metric, but sourced from SME quality) | 100% of gate invocations that include `--acknowledge-sme-risk`; log process name, finding title, and reason text | If bypass rate exceeds 15% of BLOCKER encounters, treat as a false-positive signal; pull the bypassed findings for domain expert review and add to reference dataset as labeled False Positive examples |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix (self-hosted, open-source)

Phoenix is selected because it is framework-agnostic via OpenTelemetry, requires no external service account, and can be run as a local Docker container alongside the GSD CLI during development and CI. The Claude Agent SDK emits OpenTelemetry spans when `OTEL_EXPORTER_OTLP_ENDPOINT` is set — no custom instrumentation code is required.

```bash
# Run Phoenix locally (Docker — no signup required)
docker run -p 6006:6006 arizephoenix/phoenix:latest

# Or install as a Python package and launch inline
pip install arize-phoenix
python -c "import phoenix as px; px.launch_app()"
# UI available at http://localhost:6006
```

```bash
# Enable tracing for any SME creator invocation
export OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:6006/v1/traces
export OTEL_SERVICE_NAME=gsd-sme-creator
# Then run the GSD workflow as normal — spans are emitted automatically
```

**Key Metrics to Track:**

| Metric | Source | Why It Matters |
|--------|--------|----------------|
| `total_cost_usd` per run | SDK result message `total_cost_usd` field | Primary cost signal; catch runaway analysis on unexpectedly large processes before they become expensive habits |
| `total_turns` per run | Count of tool call messages in the session stream | Proxy for analysis depth; very low turn counts (<15) suggest the agent skipped git archaeology; very high counts (>75) suggest a loop |
| `finding_counts.blocker` per process over time | SME document frontmatter `finding_counts` | Trend toward 0 suggests severity deflation (agent becoming conservative); trend upward suggests severity inflation — both require prompt review |
| Git tool call presence per session | Phoenix trace — Bash tool calls containing `git log` | Confirms CREATE-04 compliance; absence is the leading indicator of context-only analysis (Critical Failure Mode 3) |
| Fabricated path rejection rate | Count of workflow-level guardrail interventions (Section 6, fabricated file path guardrail) | Rising rate indicates hallucination regression; any rate above 5% of runs warrants model investigation |

**Alert Thresholds:**

| Condition | Threshold | Alert Action |
|-----------|-----------|--------------|
| Single run cost | > $5.00 | Log warning to console output; surface to user after completion. Not a blocker — may be a legitimately large process. |
| Single run turns | > 75 turns | Log warning; check Phoenix trace for tool call loops (repeated reads of the same file) |
| BLOCKER findings per run | > 8 BLOCKERs in one SME document | Log warning and require user confirmation before gate uses the document — high BLOCKER counts are a severity inflation signal |
| Fabricated path rejections | > 2 rejections in any 10-run window | Surface alert to developer; pull the affected sessions from Phoenix for inspection; do not auto-disable the agent |
| Monthly BLOCKER precision (flywheel) | < 70% on domain expert review sample | Switch plan-phase gate to soft-mode for that process; recalibrate agent severity prompt before restoring strict-mode |

**Smart Sampling Strategy:**

Because the SME creator is an internal developer tool invoked infrequently (once per process, refreshed after significant changes), all runs should be logged — the volume is low enough that 100% capture is practical and desirable during the first 6 months of production use.

Prioritize the following runs for immediate domain expert review (within 48 hours of completion):

1. **First run for any new process** — no historical baseline exists; expert review establishes the ground truth for that process's expected findings
2. **Any run producing `finding_counts.blocker > 0`** — BLOCKER findings gate planning; false positives are discovered immediately in the next planning session; proactive review catches them first
3. **Any run where the fabricated path guardrail fired** — indicates hallucination occurred; even if the guardrail caught it, the session trace should be inspected for partial hallucination in surviving findings
4. **Any run following an agent definition change** — prompt regressions are invisible without human review of the output

After 6 months of stable production use with < 10% expert disagreement rate on sampled findings, reduce sampling to: 100% of new-process runs and BLOCKER-producing runs; 25% random sample of refresh runs.
