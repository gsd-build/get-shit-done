# Канал Canary

dist-тег **canary** — это самый ранний preview-канал GSD. Он существует для контрибьюторов и желающих early adopter'ов, чтобы прогнать ещё не зрелые фичи на долгоживущей интеграционной ветке `dev`, до того как от них можно ожидать какой-либо стабильности.

## Политика каналов

GSD едет через три dist-тега npm, каждый питается ровно одной git-веткой. **Каналы не смешиваются.**

| Ветка | dist-тег | Аудитория | Стабильность |
|-------|----------|-----------|--------------|
| `dev` | `canary` | Контрибьюторы, желающие early adopter'ы | Best-effort. Может регрессить между сборками. Только roll-forward. |
| `main` | `next` | Мейнтейнеры, RC-тестеры | Release-candidate качество. Соблюдается bug-bar. |
| `main` | `latest` | Все остальные | Стабильный продакшен. Цель по умолчанию для `npm install`. |

`dev` — это интеграционная ветка для ещё не зрелой работы над фичами (как правило, вертикальные срезы из нескольких PR — например, дорожка MVP/TDD/UAT в 1.50.0). Когда dev-работа стабилизируется, она продвигается в `main` как RC-поезд (`vX.Y.Z-rc.N`, публикуется в `next`), и после того как RC-поезд «настоится», тот же поезд продвигается ещё раз — в `latest`.

Canary-сборка НИКОГДА не становится `next`-сборкой напрямую, а `next`-сборка НИКОГДА не становится `latest`-сборкой напрямую — каждое продвижение идёт через свежий тег и свежий релиз.

## Установка canary

```bash
# Разовый запуск (npx)
npx get-shit-done-cc@canary

# Привязать dist-тег canary глобально
npm install -g get-shit-done-cc@canary

# Привязать к точной canary-версии
npm install -g get-shit-done-cc@1.50.0-canary.1
```

Защитная очистка в CC-инсталляторе перезаписывает устаревшие блоки конфига от старых версий GSD, поэтому переустановка поверх существующего проекта безопасна.

## Когда ставить canary

✅ **Стоит** ставить canary, если ты хочешь:
- Прогнать ещё не зрелые фичи планирования/исполнения/верификации пораньше и отрепортить находки
- Убедиться, что фикс, который ты влил в `dev`, доступен сквозным потоком
- Помочь вычистить canary-bake пункты (шероховатости, которые не пойдут в `next`, пока не решены)

❌ **НЕ стоит** ставить canary на:
- Продакшен-проекты, от которых зависит доставка
- Машину, где откат означает пересоздание состояния GSD (используй профиль или workspace)
- Демо или онбординг — пиннись к `@latest`, чтобы аудитория видела стабильную поверхность

## Откат с canary

```bash
# Назад на текущий stable
npm install -g get-shit-done-cc@latest

# Или на поезд next/RC
npm install -g get-shit-done-cc@next
```

Если у тебя есть локальный проект, который взаимодействовал с canary-only фичами (например, фаза в MVP-режиме, спланированная 1.50.0-canary), артефакты планировщика в `.planning/` остаются валидны — старые версии GSD просто проигнорируют поле `**Mode:** mvp` на фазах.

## Репорт проблем по canary

Заводи issue в [трекере](https://github.com/gsd-build/get-shit-done/issues) с шаблоном `bug`. Указывай точную canary-версию (её сообщает `get-shit-done-cc --version`), чтобы triage направил репорт в поток `dev`, а не в стабильный.
