# Версионирование и стратегия релизов

GSD следует [Semantic Versioning 2.0.0](https://semver.org/) с тремя уровнями релизов, привязанными к dist-тегам npm.

## Уровни релизов

| Уровень | Что попадает | Формат версии | npm-тег | Ветка | Установка |
|---------|--------------|---------------|---------|-------|-----------|
| **Patch** | Только багфиксы | `1.27.1` | `latest` | `hotfix/1.27.1` | `npx get-shit-done-cc@latest` |
| **Minor** | Фиксы + улучшения | `1.28.0` | `latest` (после RC) | `release/1.28.0` | `npx get-shit-done-cc@next` (RC) |
| **Major** | Фиксы + улучшения + фичи | `2.0.0` | `latest` (после beta) | `release/2.0.0` | `npx get-shit-done-cc@next` (beta) |

## npm dist-теги

Всего два тега, по конвенции Angular/Next.js:

| Тег | Значение | Кем ставится |
|-----|----------|--------------|
| `latest` | Стабильный продакшен-релиз | `npm install get-shit-done-cc` (по умолчанию) |
| `next` | Пре-релиз (RC или beta) | `npm install get-shit-done-cc@next` (явный выбор) |

Строка версии (`-rc.1` против `-beta.1`) сообщает уровень стабильности. Пользователи не получают пре-релизы, пока явно их не запросят.

## Правила Semver

| Инкремент | Когда | Примеры |
|-----------|-------|---------|
| **PATCH** (1.27.x) | Багфиксы, опечатки, добавление тестов | Фикс фильтра хука, фикс порчи конфига |
| **MINOR** (1.x.0) | Не-ломающие улучшения, новые команды, поддержка новых рантаймов | Новая workflow-команда, фича discuss-режима |
| **MAJOR** (x.0.0) | Ломающие изменения формата конфига, CLI-флагов или API рантайма; новые фичи, меняющие существующее поведение | Удаление команды, смена схемы конфига |

## Прогрессия пре-релизов

Major- и minor-релизы используют разные типы пре-релизов:

```
Minor: 1.28.0-rc.1  →  1.28.0-rc.2  →  1.28.0
Major: 2.0.0-beta.1 →  2.0.0-beta.2 →  2.0.0
```

- **beta** (только major-релизы): функционал готов, но не полностью протестирован. API в основном стабильный. Сигнализирует более длинный цикл тестирования.
- **rc** (только minor-релизы): кандидат на продакшен. Ожидаются только критические фиксы.
- Каждая версия использует один тип пре-релиза на всём цикле. Действие `rc` в release-workflow автоматически выбирает правильный тип на основе версии.

## Структура веток

```
main                              ← стабильная, всегда деплоится
  │
  ├── hotfix/1.27.1               ← патч: cherry-pick фикса из main, публикация в latest
  │
  ├── release/1.28.0              ← minor: накопление фиксов + улучшений, RC-цикл
  │     ├── v1.28.0-rc.1          ← тег: публикация в next
  │     └── v1.28.0               ← тег: продвижение в latest
  │
  ├── release/2.0.0               ← major: фичи + ломающие изменения, beta-цикл
  │     ├── v2.0.0-beta.1         ← тег: публикация в next
  │     ├── v2.0.0-beta.2         ← тег: публикация в next
  │     └── v2.0.0                ← тег: продвижение в latest
  │
  ├── fix/1200-bug-description    ← ветка багфикса (мержится в main)
  ├── feat/925-feature-name       ← ветка фичи (мержится в main)
  └── chore/1206-maintenance      ← ветка обслуживания (мержится в main)
```

## Workflow-релизов

### Patch-релиз (Hotfix)

Для фиксов, которые нужно выкатить, не дожидаясь следующего minor.

Hotfix `vX.YY.Z` кумулятивно включает всё, что было в `vX.YY.{Z-1}`, плюс каждый `fix:`/`chore:` коммит, попавший в `main` с того базового тега. Base-тег — это якорь: `git cherry $BASE_TAG main` показывает ровно те коммиты, что ещё не зарелижены, а новый `vX.YY.Z` становится базой для следующего hotfix — цикл сам себя документирует.

#### Два пути

**Путь A — `hotfix.yml` (канонический, в два шага):**

1. Триггерим `hotfix.yml` с `action=create`, `version=1.27.1`, `auto_cherry_pick=true` (по умолчанию).
   - Workflow определяет `BASE_TAG` = максимальный `v1.27.*` < `v1.27.1` (то есть `1.27.1` ответвляется от `v1.27.0`; `1.27.2` ответвлялся бы от `v1.27.1`).
   - Ответвляет `hotfix/1.27.1` от `BASE_TAG`.
   - Авто-cherry-pick'ает каждый `fix:`/`chore:` коммит в `origin/main`, которого ещё нет в базе, от старого к новому. Эквивалентные патчи пропускаются через `git cherry`. `feat:`/`refactor:` **никогда** не включаются автоматически.
   - При конфликте workflow останавливается с указанием проблемного SHA. Разрешаешь руками на ветке, потом перезапускаешь finalize с `auto_cherry_pick=false`.
   - Бампит `package.json` (и `sdk/package.json`), пушит ветку и перечисляет каждый включённый SHA в summary запуска.
2. (Опционально) пушишь дополнительные ручные коммиты в `hotfix/1.27.1`.
3. Триггерим `hotfix.yml` с `action=finalize`. Workflow:
   - Прогоняет кроссплатформенный gate `install-smoke`.
   - Прогоняет полный тестовый набор + coverage.
   - Собирает SDK, бандлит `sdk-bundle/gsd-sdk.tgz` внутрь CC-tarball (паритет с `release-sdk.yml`).
   - Тегает `v1.27.1`, публикует в `@latest`, перенаправляет `@next → v1.27.1`.
   - Открывает merge-back PR в `main`.

**Путь B — `release-sdk.yml` (запасной, в один шаг):**

Активен пока npm-токен `@gsd-build/sdk` недоступен; бандлит SDK внутрь CC-tarball.

1. Триггерим `release-sdk.yml` с `action=hotfix`, `version=1.27.1`, `auto_cherry_pick=true`.
   - Job `prepare` создаёт ветку и делает cherry-pick (та же логика, что и Путь A).
   - `install-smoke` прогоняется на новой ветке.
   - Job `release` тегает, публикует в `@latest`, перенаправляет `@next`, открывает merge-back PR.
   - Идемпотентность: если `hotfix/1.27.1` уже существует (например, ты запускал `hotfix.yml create` до этого), prepare её чекаутит и перезапускает cherry-pick как no-op.
2. `dry_run=true` прогоняет полный пайплайн без пуша ветки и публикации.

### Minor-релиз (стандартный цикл)

Для накопленных фиксов и улучшений.

1. Триггерим `release.yml` с action `create` и версией (например, `1.28.0`).
2. Workflow создаёт ветку `release/1.28.0` из main, бампит package.json.
3. Триггерим `release.yml` с action `rc` чтобы опубликовать `1.28.0-rc.1` в `next`.
4. Тестируем RC: `npx get-shit-done-cc@next`.
5. Если нашли проблемы: фиксим на release-ветке, публикуем `rc.2`, `rc.3` и т.д.
6. Триггерим `release.yml` с action `finalize` — публикует `1.28.0` в `latest`.
7. Мержим release-ветку в main.

### Major-релиз

Так же как minor, но использует `-beta.N` вместо `-rc.N`, сигнализируя более длинный цикл тестирования.

1. Триггерим `release.yml` с action `create` и версией (например, `2.0.0`).
2. Триггерим `release.yml` с action `rc` чтобы опубликовать `2.0.0-beta.1` в `next`.
3. Если нашли проблемы: фиксим на release-ветке, публикуем `beta.2`, `beta.3` и т.д.
4. Триггерим `release.yml` с action `finalize` — публикует `2.0.0` в `latest`.
5. Мержим release-ветку в main.

## Conventional Commits

Имена веток мапятся на типы коммитов:

| Префикс ветки | Тип коммита | Бамп версии |
|---------------|-------------|-------------|
| `fix/` | `fix:` | PATCH |
| `feat/` | `feat:` | MINOR |
| `hotfix/` | `fix:` | PATCH (немедленный) |
| `chore/` | `chore:` | нет |
| `docs/` | `docs:` | нет |
| `refactor/` | `refactor:` | нет |

## Команды публикации (справка)

```bash
# Стабильный релиз (тег latest ставится автоматически)
npm publish

# Пре-релиз (обязательно --tag, чтобы не перетереть latest)
npm publish --tag next

# Проверить, на что указывают latest и next
npm dist-tag ls get-shit-done-cc
```
